# Ch.16 — 기업 데이터로 따라하기

> **"이 교재에서 배운 모든 것은 '개인 프로젝트'에서 검증되었다.
> 이 챕터에서는 같은 방법론이 '기업 데이터'에서도 동일하게 작동하는지 증명한다."**

Part 1~3에서 우리는 IT 도메인 2,392개 문서를 Neo4j 지식그래프로 변환하는 전 과정을 경험했습니다.
이 챕터에서는 **기업에서 흔히 만나는 4가지 시나리오**에 같은 워크플로우를 적용합니다.

> 📌 **이 챕터의 목적**
>
> "우리 프로젝트만의 특수한 방법론"이 아니라,
> **범용적으로 재사용 가능한 그래프 구축 프레임워크**임을 보여주는 것.
> 각 시나리오에서 변하는 것과 변하지 않는 것을 명확히 구분합니다.

## 16.1 8단계 워크플로우의 범용성

### 16.1.1 워크플로우 리마인드

```
STEP 1: 인프라 구성       → Docker + 플러그인
STEP 2: 데이터 수집       → 원본 데이터 파싱 + 정량 분석
STEP 3: 스키마 설계       → QDM + 노드/관계/속성 결정
STEP 4: 표본 적재         → 소량 데이터로 스키마 검증
STEP 5: 스키마 검증+수정  → 5계층 진단 + 수정 + 재검증
STEP 6: 확장 적재         → 더 많은 데이터로 확장
STEP 7: 전체 적재+자동화  → 파이프라인 자동화
STEP 8: 활용              → GraphRAG, 분석, 시각화
```

### 16.1.2 변하지 않는 것 vs 달라지는 것

| 항목 | 변하지 않는 것 (프레임워크) | 달라지는 것 (도메인) |
|------|---------------------------|---------------------|
| **STEP 1** | Docker 기반 Neo4j 구성 | 에디션 선택, 메모리 설정, 보안 구성 |
| **STEP 2** | 정량 분석 → 스키마 근거 확보 | 데이터 소스 (RDB, API, CSV, 문서) |
| **STEP 3** | QDM 방법론 (질문→패턴→스키마) | 핵심 질문의 내용, 노드/관계 이름 |
| **STEP 4** | 표본 10~50건으로 빠른 검증 | 적재 대상 데이터 |
| **STEP 5** | 5계층 검증 프레임워크 | 도메인 특화 검증 규칙 |
| **STEP 6~7** | 반복 순환 + 배치 자동화 | 데이터 규모, ETL 도구 |
| **STEP 8** | 그래프 활용 패턴 | 활용 목적 (검색, 분석, 추천) |

> [설계 원칙] **"방법론은 동일하다. 달라지는 것은 '질문'과 '데이터'뿐이다."**
>
> 이것이 Question-Driven Modeling의 핵심입니다.
> 어떤 도메인이든 "이 데이터로 답변하고 싶은 질문"을 먼저 정의하면,
> 스키마는 자연스럽게 도출됩니다.

### 16.1.3 기업 환경과 개인 프로젝트의 실질적 차이

그러나 기업 환경에는 개인 프로젝트에 없는 **현실적 제약**이 있습니다:

| 제약 | 개인 프로젝트 | 기업 환경 |
|------|-------------|-----------|
| 데이터 접근 | 파일 시스템에서 자유롭게 | 권한 신청, 개인정보 마스킹, 반출 절차 |
| 스키마 변경 | 자유롭게 수정 | 변경 관리 프로세스, 영향 분석, 승인 |
| 이해관계자 | 본인 1명 | 데이터 오너, DBA, 개발팀, 비즈니스 |
| 품질 기준 | 스스로 판단 | SLA, 감사, 컴플라이언스 |
| 인프라 | 로컬 Docker | 프로덕션 클러스터, 백업, 모니터링 |

이런 차이를 고려하여 각 시나리오를 설계합니다.

## 16.2 시나리오 A: 제품 카탈로그 → 지식그래프

### 16.2.1 비즈니스 맥락

> "우리 회사는 3,000개 제품을 판매하는 B2B 기업입니다.
> 영업팀이 '이 제품과 호환되는 부품은?', '경쟁사 대비 우위 기술은?'
> 같은 질문에 빠르게 답변하길 원합니다."

**데이터 소스:**
- ERP의 제품 마스터 테이블 (CSV 추출)
- 기술 사양서 (PDF/DOCX)
- 영업팀 FAQ 문서

### 16.2.2 STEP 3 적용: QDM으로 스키마 도출

**핵심 질문 5개:**

| # | 질문 | 답변 패턴 | 도출되는 스키마 |
|---|------|-----------|----------------|
| Q1 | "이 제품과 호환되는 부품은?" | 제품 → 부품 목록 | `(:Product)-[:COMPATIBLE_WITH]->(:Part)` |
| Q2 | "이 기술을 사용하는 제품군은?" | 기술 → 제품 목록 | `(:Product)-[:USES_TECH]->(:Technology)` |
| Q3 | "A 제품과 B 제품의 기술 차이는?" | 제품 A/B 속성 비교 | `(:Product)-[:COMPARED_WITH]->(:Product)` |
| Q4 | "이 카테고리의 베스트셀러는?" | 카테고리 → 제품 랭킹 | `(:Product)-[:BELONGS_TO]->(:Category)` |
| Q5 | "고객 X가 관심 가질 제품은?" | 고객 구매 이력 기반 | `(:Customer)-[:PURCHASED]->(:Product)` |

**도출된 스키마:**

```
(:Product) -[:BELONGS_TO]-> (:Category)
(:Product) -[:USES_TECH]-> (:Technology)
(:Product) -[:COMPATIBLE_WITH]-> (:Part)
(:Product) -[:COMPARED_WITH]-> (:Product)
(:Customer) -[:PURCHASED]-> (:Product)
(:Product) -[:MANUFACTURED_BY]-> (:Supplier)
```

**노드 설계:**

```cypher
// 제약조건
CREATE CONSTRAINT product_id IF NOT EXISTS
FOR (n:Product) REQUIRE n.id IS UNIQUE;

CREATE CONSTRAINT category_id IF NOT EXISTS
FOR (n:Category) REQUIRE n.id IS UNIQUE;

CREATE CONSTRAINT part_id IF NOT EXISTS
FOR (n:Part) REQUIRE n.id IS UNIQUE;

// 시드 노드 (카테고리 계층)
MERGE (c:Category {id: 'sensors'})
SET c.name = '센서류', c.level = 1;

MERGE (c:Category {id: 'temp_sensors'})
SET c.name = '온도 센서', c.level = 2;

// 관계 생성 — 양쪽 노드를 각각 MERGE 후 관계 MERGE
MERGE (child:Category {id: 'temp_sensors'})
MERGE (parent:Category {id: 'sensors'})
MERGE (child)-[:SUBCLASS_OF]->(parent);
```

> ⚠️ **전체 패턴 MERGE 금지**: `MERGE (a)-[:REL]->(b)`를 한 줄로 쓰면,
> 패턴 전체가 없을 때 노드까지 새로 생성하여 중복 노드가 발생합니다.
> 반드시 **노드를 각각 MERGE한 후** 관계를 MERGE하세요.
> 이것은 Ch.02에서 배운 핵심 규칙입니다.

### 16.2.3 STEP 4 적용: 표본 적재

```cypher
// ERP CSV에서 제품 10건 적재
LOAD CSV WITH HEADERS FROM 'file:///products_sample.csv' AS row
MERGE (p:Product {id: row.product_id})
SET p.name = row.product_name,
    p.sku = row.sku,
    p.price = toFloat(row.price),
    p.launch_date = date(row.launch_date)

// 카테고리 연결
WITH p, row
MERGE (c:Category {id: row.category_id})
MERGE (p)-[:BELONGS_TO]->(c)
```

### 16.2.4 STEP 5 적용: 검증

```cypher
// L1: 구조 검증
SHOW CONSTRAINTS;
CALL apoc.meta.stats() YIELD nodeCount, relCount RETURN nodeCount, relCount;

// L2: 필수 속성 확인
MATCH (p:Product)
WHERE p.name IS NULL OR p.sku IS NULL
RETURN p.id, 'name 또는 sku 누락' AS 이슈;

// L3: BELONGS_TO 누락
MATCH (p:Product)
WHERE NOT EXISTS { (p)-[:BELONGS_TO]->(:Category) }
RETURN p.id, p.name AS '카테고리 미연결 제품';

// L4: 고립 Part 노드 (어떤 제품과도 연결되지 않은 부품)
MATCH (pt:Part)
WHERE NOT EXISTS { (:Product)-[:COMPATIBLE_WITH]->(pt) }
RETURN pt.id, pt.name AS '미연결 부품';
```

### 16.2.5 이 시나리오의 핵심 교훈

| 우리 프로젝트와의 대응 | 제품 카탈로그 |
|----------------------|-------------|
| Concept | Product |
| Domain | Category |
| SUBCLASS_OF | 카테고리 계층 |
| BELONGS_TO | 제품 → 카테고리 |
| MENTIONED_IN | 제품 → 사양서(Document) |
| Top 50 시드 | 주요 카테고리 + 핵심 기술 |

> 💡 **QDM은 도메인에 독립적입니다.**
> "딥러닝의 하위 기술은?"이라는 질문과 "센서류의 하위 제품군은?"이라는 질문은
> 구조적으로 동일합니다. 둘 다 `SUBCLASS_OF` 계층 탐색입니다.

## 16.3 시나리오 B: IT 인프라 CMDB → 의존성 그래프

### 16.3.1 비즈니스 맥락

> "인프라팀이 서버 장애 시 '영향받는 서비스는?', '이 서버가 죽으면
> 어떤 고객이 영향받는가?'에 즉시 답변해야 합니다.
> 현재 Excel CMDB로는 의존성 추적이 불가능합니다."

**왜 그래프가 최적인가:**

CMDB(Configuration Management Database)는 본질적으로 **의존성 네트워크**입니다.
서버 A가 데이터베이스 B에 의존하고, 서비스 C가 서버 A를 통해 제공되며,
고객 D가 서비스 C를 이용합니다. RDB에서 이 의존성을 추적하려면
4단계 JOIN이 필요하지만, 그래프에서는 가변 길이 경로 한 줄입니다.

### 16.3.2 QDM 핵심 질문

| # | 질문 | Cypher 패턴 |
|---|------|------------|
| Q1 | "서버 X가 죽으면 영향받는 서비스는?" | `(s:Server {id:'X'})<-[:RUNS_ON*1..3]-(svc:Service)` |
| Q2 | "이 서비스의 전체 의존성 트리는?" | `(svc:Service)-[:DEPENDS_ON*1..5]->(dep)` |
| Q3 | "단일 장애점(SPOF)인 서버는?" | 들어오는 관계가 3개 이상인 Server |
| Q4 | "이 DB를 패치하면 재시작 필요한 서비스는?" | `(db:Database)<-[:CONNECTS_TO]-(app)-[:PROVIDES]->(svc)` |
| Q5 | "고객 X에게 영향을 주는 모든 인프라는?" | `(c:Customer)-[:USES]->(svc)-[:RUNS_ON|DEPENDS_ON*1..5]->(infra)` |

### 16.3.3 스키마 설계

```
(:Server)    -[:HOSTS]->        (:Application)
(:Application) -[:CONNECTS_TO]-> (:Database)
(:Application) -[:PROVIDES]->    (:Service)
(:Service)   -[:DEPENDS_ON]->   (:Service)
(:Customer)  -[:USES]->         (:Service)
(:Server)    -[:LOCATED_IN]->   (:DataCenter)
(:Server)    -[:BELONGS_TO]->   (:ServerGroup)
```

**노드 속성 설계:**

```cypher
MERGE (s:Server {id: 'srv_web_01'})
SET s.hostname = 'web-prod-01.company.com',
    s.ip = '10.0.1.101',
    s.os = 'Ubuntu 22.04',
    s.cpu_cores = 8,
    s.memory_gb = 32,
    s.status = 'running',
    s.environment = 'production',
    s.owner = 'infra_team'
```

### 16.3.4 CMDB 특화 검증

```cypher
// SPOF 탐지 — 3개 이상 서비스가 의존하는 서버
MATCH (s:Server)<-[:RUNS_ON]-(app)-[:PROVIDES]->(svc)
WITH s, count(DISTINCT svc) AS service_count
WHERE service_count >= 3
RETURN s.hostname AS SPOF후보,
       s.environment AS 환경,
       service_count AS 의존서비스수
ORDER BY service_count DESC

// 장애 영향 분석 — 특정 서버 장애 시 영향 범위
MATCH path = (s:Server {id: 'srv_db_01'})<-[:CONNECTS_TO|RUNS_ON*1..4]-(affected)
RETURN DISTINCT labels(affected) AS 유형,
       affected.id AS 영향대상,
       length(path) AS 간접도
ORDER BY 간접도
```

### 16.3.5 RDB와의 결정적 차이

```sql
-- SQL: "서버 X 장애 시 영향받는 고객" (4단계 JOIN)
SELECT DISTINCT c.name
FROM servers s
JOIN applications a ON a.server_id = s.id
JOIN services svc ON svc.app_id = a.id
JOIN customer_services cs ON cs.service_id = svc.id
JOIN customers c ON c.id = cs.customer_id
WHERE s.hostname = 'srv_db_01';
```

```cypher
// Cypher: 동일한 질문 (가변 길이 경로 1줄)
MATCH (s:Server {id: 'srv_db_01'})<-[*1..4]-(c:Customer)
RETURN DISTINCT c.name AS 영향받는고객
```

> 💡 **의존성 깊이가 변하면?** SQL은 JOIN 단계를 수동으로 추가해야 합니다.
> Cypher는 `*1..4`를 `*1..6`으로 바꾸는 것만으로 끝납니다.
> CMDB처럼 의존성 깊이가 가변적인 데이터는 그래프의 결정적 우위 영역입니다.

## 16.4 시나리오 C: 고객 여정 → 행동 그래프

### 16.4.1 비즈니스 맥락

> "마케팅팀이 '어떤 경로로 구매에 이르는 고객이 가장 많은가?',
> '이탈 고객의 공통 행동 패턴은?'을 분석하고 싶어합니다.
> 현재 Google Analytics와 CRM 데이터가 분리되어 있어
> 통합 여정 분석이 어렵습니다."

### 16.4.2 QDM 핵심 질문

| # | 질문 | 분석 목적 |
|---|------|-----------|
| Q1 | "가장 흔한 구매 경로 Top 5는?" | 전환 최적화 |
| Q2 | "이탈 고객의 마지막 3단계 행동은?" | 이탈 방지 |
| Q3 | "캠페인 A를 본 고객이 가장 많이 하는 행동은?" | 캠페인 효과 측정 |
| Q4 | "VIP 고객과 일반 고객의 행동 패턴 차이는?" | 세그먼트 이해 |
| Q5 | "특정 페이지를 본 후 구매까지 평균 몇 단계인가?" | UX 최적화 |

### 16.4.3 스키마 설계

```
(:Customer)  -[:PERFORMED {timestamp, session}]->  (:Event)
(:Event)     -[:NEXT]->                            (:Event)
(:Event)     -[:ON_PAGE]->                         (:Page)
(:Customer)  -[:BELONGS_TO]->                      (:Segment)
(:Campaign)  -[:TARGETS]->                         (:Segment)
(:Customer)  -[:EXPOSED_TO]->                      (:Campaign)
```

**이벤트 체인 모델링 — 왜 이렇게 설계하는가:**

고객 여정은 **시간 순서가 있는 이벤트 체인**입니다.
각 이벤트를 노드로 모델링하고, `NEXT` 관계로 순서를 연결하면
"가장 흔한 경로"를 패턴 매칭으로 찾을 수 있습니다.

```cypher
// 이벤트 체인 적재
MERGE (c:Customer {id: 'cust_001'})

MERGE (e1:Event {id: 'evt_001_001'})
SET e1.type = 'page_view', e1.timestamp = datetime('2025-03-01T10:00:00')
MERGE (c)-[:PERFORMED]->(e1)

MERGE (e2:Event {id: 'evt_001_002'})
SET e2.type = 'add_to_cart', e2.timestamp = datetime('2025-03-01T10:05:00')
MERGE (c)-[:PERFORMED]->(e2)
MERGE (e1)-[:NEXT]->(e2)

MERGE (e3:Event {id: 'evt_001_003'})
SET e3.type = 'purchase', e3.timestamp = datetime('2025-03-01T10:10:00')
MERGE (c)-[:PERFORMED]->(e3)
MERGE (e2)-[:NEXT]->(e3)
```

### 16.4.4 여정 분석 쿼리

```cypher
// 가장 흔한 3-스텝 구매 경로
MATCH (e1:Event)-[:NEXT]->(e2:Event)-[:NEXT]->(e3:Event {type: 'purchase'})
RETURN e1.type + ' → ' + e2.type + ' → ' + e3.type AS 경로,
       count(*) AS 빈도
ORDER BY 빈도 DESC
LIMIT 5

// 이탈 고객의 마지막 행동 분석
MATCH (c:Customer)-[:BELONGS_TO]->(:Segment {id: 'churned'})
MATCH (c)-[:PERFORMED]->(last:Event)
WHERE NOT (last)-[:NEXT]->()    // 마지막 이벤트
RETURN last.type AS 마지막행동, count(*) AS 이탈수
ORDER BY 이탈수 DESC

// 캠페인 노출 후 구매까지 평균 단계 수
MATCH (c:Customer)-[:EXPOSED_TO]->(:Campaign {id: 'spring_2025'})
MATCH path = (first:Event)-[:NEXT*1..10]->(purchase:Event {type: 'purchase'})
WHERE (c)-[:PERFORMED]->(first)
RETURN avg(length(path)) AS 평균단계, count(*) AS 구매건수
```

### 16.4.5 고객 여정 특화 고려사항

> ⚠️ **이벤트 노드 폭발 주의**
>
> 고객 10만 명 × 평균 이벤트 50개 = **500만 이벤트 노드**.
> 이벤트를 모두 개별 노드로 만들면 그래프가 급격히 커집니다.
>
> **대응 전략:**
> - 핵심 이벤트만 노드화 (page_view, add_to_cart, purchase, churn)
> - 세부 이벤트는 관계 속성으로 처리
> - 시간 기반 파티셔닝 (월별 서브그래프)
> - 집계 노드 활용 (일별 행동 요약)

## 16.5 시나리오 D: 법률/규제 → 컴플라이언스 그래프

### 16.5.1 비즈니스 맥락

> "법무팀이 '이 규정이 바뀌면 우리 회사의 어떤 프로세스가 영향받는가?',
> '이 인증을 받으려면 어떤 통제 항목을 충족해야 하는가?'에
> 신속하게 답변해야 합니다. 현재 수백 개 규정이 Excel로 관리되고 있어
> 교차 영향 분석이 사실상 불가능합니다."

### 16.5.2 QDM 핵심 질문

| # | 질문 | 도출되는 스키마 |
|---|------|----------------|
| Q1 | "규정 X가 요구하는 통제 항목은?" | `(:Regulation)-[:REQUIRES]->(:Control)` |
| Q2 | "통제 항목 Y를 이행하는 프로세스는?" | `(:Control)-[:IMPLEMENTED_BY]->(:Process)` |
| Q3 | "규정 X가 변경되면 영향받는 프로세스는?" | 가변 길이 경로 탐색 |
| Q4 | "ISO 27001과 GDPR의 공통 통제 항목은?" | 두 규정의 교집합 |
| Q5 | "미이행 통제 항목은?" | IMPLEMENTED_BY 누락 검출 |

### 16.5.3 스키마 설계

```
(:Regulation)  -[:REQUIRES]->        (:Control)
(:Control)     -[:IMPLEMENTED_BY]->  (:Process)
(:Regulation)  -[:SUPERSEDES]->      (:Regulation)    // 상위법/하위법
(:Control)     -[:MAPS_TO]->         (:Control)       // 교차 매핑
(:Process)     -[:OWNED_BY]->        (:Department)
(:Audit)       -[:EVALUATES]->       (:Control)
(:Audit)       -[:FINDING]->         (:NonCompliance)
```

**우리 프로젝트와의 대응:**

| 우리 프로젝트 | 컴플라이언스 그래프 |
|--------------|---------------------|
| Concept | Control (통제 항목) |
| Standard (ISO 27001) | Regulation |
| SUBCLASS_OF | 규정 계층 (법률→시행령→시행규칙) |
| DEFINED_BY | REQUIRES (규정이 통제 항목을 정의) |
| BELONGS_TO (Domain) | OWNED_BY (부서 소속) |
| MENTIONED_IN (Document) | EVALUATES (감사 연결) |

### 16.5.4 컴플라이언스 특화 쿼리

```cypher
// 규정 변경 영향 분석: "개인정보보호법 개정 시 영향받는 프로세스"
MATCH (reg:Regulation {id: 'pipa_2025'})-[:REQUIRES]->(ctrl:Control)
      -[:IMPLEMENTED_BY]->(proc:Process)
RETURN ctrl.name AS 통제항목,
       proc.name AS 영향프로세스,
       proc.owner AS 담당부서
ORDER BY proc.owner

// ISO 27001과 GDPR의 공통 통제 항목
MATCH (iso:Regulation {id: 'iso_27001'})-[:REQUIRES]->(ctrl:Control)
      <-[:REQUIRES]-(gdpr:Regulation {id: 'gdpr'})
RETURN ctrl.id, ctrl.name AS 공통통제항목,
       ctrl.description
ORDER BY ctrl.id

// 미이행 통제 항목 (갭 분석)
MATCH (reg:Regulation {id: 'iso_27001'})-[:REQUIRES]->(ctrl:Control)
WHERE NOT EXISTS { (ctrl)-[:IMPLEMENTED_BY]->(:Process) }
RETURN ctrl.id AS 미이행항목,
       ctrl.name AS 통제항목명,
       ctrl.priority AS 우선순위
ORDER BY ctrl.priority DESC

// 감사 결과 연결
MATCH (audit:Audit {id: 'audit_2025_q1'})-[:FINDING]->(nc:NonCompliance)
      -[:VIOLATES]->(ctrl:Control)<-[:REQUIRES]-(reg:Regulation)
RETURN reg.name AS 관련규정,
       ctrl.name AS 위반통제항목,
       nc.severity AS 심각도,
       nc.remediation_deadline AS 시정기한
ORDER BY nc.severity DESC
```

> 💡 **컴플라이언스 그래프의 진정한 가치**
>
> Excel에서는 "GDPR 제17조가 변경되면 우리 회사의 어떤 프로세스가 영향받는가?"에
> 답하려면 여러 시트를 수동으로 교차 참조해야 합니다.
> 그래프에서는 `MATCH path = (reg)-[:REQUIRES*1..3]->(affected)` 한 줄이면 됩니다.
>
> 규정이 수백 개이고 통제 항목이 수천 개인 환경에서,
> 교차 매핑과 영향 분석의 속도 차이는 **시간 단위에서 초 단위**로 줄어듭니다.

## 16.6 공통 패턴과 차이점 분석

### 16.6.1 4개 시나리오의 공통 구조

모든 시나리오에서 동일하게 나타나는 **5가지 범용 패턴**:

| 패턴 | 우리 프로젝트 | 제품 카탈로그 | CMDB | 고객 여정 | 컴플라이언스 |
|------|-------------|-------------|------|-----------|-------------|
| **계층 분류** | Concept→Domain | Product→Category | Server→DataCenter | Customer→Segment | Control→Regulation |
| **상하위 관계** | SUBCLASS_OF | SUBCLASS_OF | DEPENDS_ON | NEXT | SUPERSEDES |
| **추적/연결** | MENTIONED_IN | 사양서 연결 | RUNS_ON | PERFORMED | EVALUATES |
| **비교/매핑** | COMPARED_WITH | 제품 비교 | (없음) | (없음) | MAPS_TO |
| **누락 검출** | BELONGS_TO 누락 | 카테고리 미연결 | SPOF 탐지 | 이탈 감지 | 미이행 갭 |

### 16.6.2 스키마 설계 판단 기준 — 범용 결정 트리

어떤 도메인이든 다음 결정 트리를 적용하면 노드와 관계를 결정할 수 있습니다:

```
이 엔티티가...

① 자체적인 관계를 가지는가?
   ├─ YES → 노드로 분리
   └─ NO → ②로

② 다른 엔티티에서도 참조되는가?
   ├─ YES → 노드로 분리
   └─ NO → ③으로

③ 검색/필터링 대상인가?
   ├─ YES → 노드로 분리 (인덱스 활용)
   └─ NO → 속성으로 처리

④ 시간에 따라 변하는 값인가?
   ├─ YES → 이벤트 노드 또는 관계 속성
   └─ NO → 노드/속성 중 선택

⑤ 수량이 폭발할 수 있는가?
   ├─ YES → 집계 노드 또는 관계 속성으로 압축
   └─ NO → 개별 노드 OK
```

### 16.6.3 공통 안티패턴

> ⚠️ **[안티패턴] 4개 시나리오에서 공통으로 발견되는 실수**
>
> **1. God Node (슈퍼 노드)**
> - 하나의 노드에 수천 개의 관계가 집중
> - 예: 모든 제품이 하나의 "전자제품" 카테고리에 연결
> - 대응: 카테고리 세분화, 중간 계층 추가
>
> **2. 과도한 비정규화**
> - RDB 사고방식으로 속성에 모든 정보를 넣음
> - 예: Product 노드에 `supplier_name`, `supplier_address`, `supplier_phone` 속성
> - 대응: Supplier를 별도 노드로 분리
>
> **3. 관계 타입 부족**
> - 모든 관계를 `RELATED_TO`로 퉁침
> - 예: 제품-부품, 제품-기술, 제품-카테고리 모두 `RELATED_TO`
> - 대응: 의미가 다르면 관계 타입이 달라야 함
>
> **4. id 설계 소홀**
> - 자동 생성 UUID에만 의존, 의미 있는 자연키 미사용
> - 예: `{id: 'a3f7...'}` vs `{id: 'srv-web-01'}`
> - 대응: 도메인에서 자연스러운 식별자가 있으면 그것을 id로

### 16.6.4 규모별 기술 선택 가이드

| 규모 | 노드+관계 수 | 적재 방식 | 인프라 | 주의사항 |
|------|-------------|-----------|--------|----------|
| **소규모** | ~10만 | Cypher MERGE | 단일 Docker | 개인 프로젝트 수준 |
| **중규모** | 10만~1,000만 | LOAD CSV + 배치 | 단일 서버, 충분한 메모리 | 인덱스 최적화 필수 |
| **대규모** | 1,000만~1억 | neo4j-admin import | 클러스터, 전용 서버 | 초기 적재 시 대량 임포트, 이후 증분 |
| **초대규모** | 1억+ | Spark Connector, Kafka | 클러스터 + 샤딩 | 아키텍처 전문가 필요 |

```cypher
// 대량 초기 적재: neo4j-admin import (cypher-shell이 아님!)
// 이 명령어는 Neo4j가 중지된 상태에서 PowerShell에서 실행합니다
// neo4j-admin database import full \
//   --nodes=Product=import/products.csv \
//   --nodes=Category=import/categories.csv \
//   --relationships=BELONGS_TO=import/product_category.csv \
//   neo4j
```

> 💡 **neo4j-admin import vs LOAD CSV**
>
> | 항목 | neo4j-admin import | LOAD CSV |
> |------|-------------------|----------|
> | 속도 | 초당 수백만 건 | 초당 수천 건 |
> | 조건 | Neo4j 중지 상태 | Neo4j 실행 중 |
> | 용도 | 초기 대량 적재 | 증분 적재, 일상 업데이트 |
> | 중복 처리 | 불가 (중복 시 에러) | MERGE로 멱등 처리 |
>
> 대규모 기업 데이터는 초기 적재를 `neo4j-admin import`로 수행하고,
> 이후 변경분만 LOAD CSV 또는 Python Driver로 증분 적재하는 것이 일반적입니다.

### 목표

자신의 업무에서 "그래프로 답변하고 싶은 질문"을 3개 적으세요.

1. "김부장님의 직속 보고 라인은 누구까지인가?"
2. "프로젝트 A에 참여 중인 팀원의 보유 스킬은?"
3. "이 스킬을 가진 가용 인력은 누구인가?"

### ② QDM 적용 — 질문에서 스키마 도출 (5분)

각 질문에서 노드(명사)와 관계(동사)를 추출하세요:

```
Q1 → (:Employee)-[:REPORTS_TO]->(:Employee)
Q2 → (:Employee)-[:PARTICIPATES_IN]->(:Project)
     (:Employee)-[:HAS_SKILL]->(:Skill)
Q3 → (:Employee)-[:HAS_SKILL]->(:Skill)
     + Employee.availability 속성
```

### ③ 표본 적재 (5분)

노드 5~10개, 관계 10~15개를 MERGE로 적재하세요:

```cypher
// 예시: HR 미니 온톨로지
MERGE (e1:Employee {id: 'kim_bj'})
SET e1.name = '김부장', e1.department = '개발팀', e1.level = '부장';

MERGE (e2:Employee {id: 'lee_cj'})
SET e2.name = '이차장', e2.department = '개발팀', e2.level = '차장';

MERGE (e1)<-[:REPORTS_TO]-(e2);

MERGE (s:Skill {id: 'python'})
SET s.name = 'Python', s.category = 'programming';

MERGE (e2)-[:HAS_SKILL {level: 'advanced'}]->(s);

MERGE (p:Project {id: 'prj_alpha'})
SET p.name = '프로젝트 알파', p.status = 'active';

MERGE (e2)-[:PARTICIPATES_IN]->(p);
```

### ④ 검증 — 질문에 답변할 수 있는가? (5분)

처음 정의한 3개 질문을 Cypher로 실행하세요:

```cypher
// Q1: 김부장님의 보고 라인
MATCH path = (kim:Employee {id: 'kim_bj'})<-[:REPORTS_TO*1..5]-(subordinate)
RETURN [n IN nodes(path) | n.name] AS 보고라인

// Q2: 프로젝트 알파 팀원의 스킬
MATCH (e:Employee)-[:PARTICIPATES_IN]->(:Project {id: 'prj_alpha'})
OPTIONAL MATCH (e)-[:HAS_SKILL]->(s:Skill)
RETURN e.name AS 팀원, collect(s.name) AS 보유스킬

// Q3: Python 스킬 보유 가용 인력
MATCH (e:Employee)-[:HAS_SKILL]->(:Skill {id: 'python'})
WHERE e.availability <> 'unavailable' OR e.availability IS NULL
RETURN e.name, e.department, e.level
```

## 16.7 기존 RDB에서 Neo4j로 마이그레이션

기업 환경에서 가장 빈번한 시나리오는 **이미 RDB에 있는 데이터**를 Neo4j로 가져오는 것입니다.
"데이터를 처음부터 모으는 것"(우리 프로젝트)과 달리, 이미 정규화된 테이블 구조를
**역정규화(denormalize)하여 그래프로 변환**해야 합니다.

### 16.7.1 RDB→Graph 변환의 핵심 원칙

```
┌──────────────────────────────────────────────────────────┐
│ RDB 요소             →  Graph 요소                       │
├──────────────────────────────────────────────────────────┤
│ 테이블 (엔티티)       →  노드 (:Label)                   │
│ 행 (레코드)           →  개별 노드 인스턴스               │
│ 컬럼 (속성)           →  노드 속성 {key: value}          │
│ 외래키 (FK)           →  관계 -[:TYPE]->                 │
│ JOIN 테이블           →  직접 관계 또는 중간 노드         │
│ 자기 참조 FK          →  자기 참조 관계 (계층)            │
│ ENUM/코드 테이블      →  속성 또는 분류 노드              │
└──────────────────────────────────────────────────────────┘
```

### 16.7.2 실전 예시: HR 테이블 → 그래프

**AS-IS (RDB):**

```sql
-- employees 테이블
CREATE TABLE employees (
  id INT PRIMARY KEY,
  name VARCHAR(100),
  department_id INT REFERENCES departments(id),
  manager_id INT REFERENCES employees(id)  -- 자기 참조
);

-- departments 테이블
CREATE TABLE departments (
  id INT PRIMARY KEY,
  name VARCHAR(100)
);

-- employee_skills 테이블 (JOIN 테이블)
CREATE TABLE employee_skills (
  employee_id INT REFERENCES employees(id),
  skill_id INT REFERENCES skills(id),
  level VARCHAR(20),
  PRIMARY KEY (employee_id, skill_id)
);
```

**TO-BE (Neo4j):**

```cypher
// 1. 엔티티 테이블 → 노드
LOAD CSV WITH HEADERS FROM 'file:///departments.csv' AS row
MERGE (d:Department {id: toInteger(row.id)})
SET d.name = row.name;

LOAD CSV WITH HEADERS FROM 'file:///employees.csv' AS row
MERGE (e:Employee {id: toInteger(row.id)})
SET e.name = row.name;

// 2. 외래키 → 관계
LOAD CSV WITH HEADERS FROM 'file:///employees.csv' AS row
MATCH (e:Employee {id: toInteger(row.id)})
MATCH (d:Department {id: toInteger(row.department_id)})
MERGE (e)-[:WORKS_IN]->(d);

// 3. 자기 참조 FK → 계층 관계
LOAD CSV WITH HEADERS FROM 'file:///employees.csv' AS row
WHERE row.manager_id IS NOT NULL
MATCH (e:Employee {id: toInteger(row.id)})
MATCH (m:Employee {id: toInteger(row.manager_id)})
MERGE (e)-[:REPORTS_TO]->(m);

// 4. JOIN 테이블 → 관계 (속성 포함)
LOAD CSV WITH HEADERS FROM 'file:///employee_skills.csv' AS row
MATCH (e:Employee {id: toInteger(row.employee_id)})
MATCH (s:Skill {id: toInteger(row.skill_id)})
MERGE (e)-[:HAS_SKILL {level: row.level}]->(s);
```

### 16.7.3 마이그레이션 3단계 프로세스

```
STEP 1: 스키마 매핑 (1~2일)
───────────────────────────
- RDB 스키마(ERD) 분석
- 테이블→노드, FK→관계 매핑 테이블 작성
- JOIN 테이블의 처리 방식 결정 (관계 속성 vs 중간 노드)
- QDM 적용: 핵심 질문 도출 → 그래프 스키마 설계

STEP 2: 데이터 추출 + 적재 (2~3일)
───────────────────────────
- RDB에서 CSV 내보내기
- LOAD CSV + MERGE로 Neo4j에 적재
- 배치 단위: CALL IN TRANSACTIONS OF 1000 ROWS

STEP 3: 검증 + 동기화 전략 (1~2일)
───────────────────────────
- 5계층 검증 프레임워크 적용 (Ch.09)
- RDB와의 동기화 방식 결정:
  ① 일회성 마이그레이션 (RDB 폐기)
  ② 주기적 동기화 (ETL/CDC)
  ③ 병행 운영 (RDB = OLTP, Neo4j = 분석/탐색)
```

### 16.7.4 마이그레이션 시 흔한 실수

> ⚠️ **[안티패턴] RDB 구조를 그대로 옮기기**
>
> ```cypher
> // ❌ RDB의 JOIN 테이블을 그대로 노드로 만듦
> (:EmployeeSkill {employee_id: 1, skill_id: 5, level: 'advanced'})
>
> // ✅ 관계로 변환 (관계 속성에 level 포함)
> (:Employee)-[:HAS_SKILL {level: 'advanced'}]->(:Skill)
> ```
>
> RDB의 JOIN 테이블은 그래프에서 **관계**로 표현해야 합니다.
> JOIN 테이블을 노드로 만들면 RDB의 단점을 그대로 가져오는 꼴입니다.
> 단, JOIN 테이블에 속성이 3개 이상이면 **중간 노드**가 적합할 수 있습니다.

> ⚠️ **[안티패턴] 모든 컬럼을 속성으로 복사**
>
> RDB의 모든 컬럼을 그대로 노드 속성으로 복사하면 **속성 비대(Property Bloat)**가 됩니다.
> 그래프에서 **질문에 답하는 데 필요한 속성만** 선별하세요.
> 나머지는 RDB에 두고 ID로 연결하면 됩니다.
