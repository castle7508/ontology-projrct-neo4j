# Ch.17 — 프로덕션 운영과 거버넌스

> **"그래프를 만드는 것은 시작이다. 그래프를 '살아있게' 유지하는 것이 진짜 과제다."**

Ch.16에서 기업 데이터로 온톨로지를 구축하는 방법을 배웠습니다.
이 챕터에서는 구축된 그래프를 **프로덕션 환경에서 안전하고 지속적으로 운영**하기 위한
아키텍처, 거버넌스, 보안, 성능 관리 전략을 다룹니다.

> 📌 **왜 이 챕터가 필요한가**
>
> 개인 프로젝트에서는 Docker 하나로 충분하지만,
> 기업 환경에서는 다음 질문에 답해야 합니다:
>
> - "서버가 죽으면 데이터는 어떻게 되나요?"
> - "누가 스키마를 변경할 수 있나요?"
> - "쿼리가 느려지면 어떻게 알 수 있나요?"
> - "민감한 속성을 특정 팀에만 보이게 할 수 있나요?"

## 17.1 프로덕션 아키텍처

### 17.1.1 단일 인스턴스 vs 클러스터

| 구성 | 적합한 환경 | 가용성 |
|------|-----------|--------|
| **단일 인스턴스** | 개발/PoC, 소규모 서비스 | 서버 1대 장애 = 서비스 중단 |
| **Primary-Secondary** | 읽기 부하 분산 | 읽기 가용, 쓰기는 Primary에 집중 |
| **클러스터 (3+ 노드)** | 프로덕션, 고가용성 필수 | 1~2대 장애에도 서비스 유지 |

**Neo4j 5+ 클러스터 구성 (Enterprise Edition):**

Neo4j 5에서는 **Raft 기반 합의 프로토콜**을 사용합니다.
3대 이상의 서버가 클러스터를 구성하고, 쓰기 요청은 리더 노드가 처리하며,
나머지 노드에 자동으로 복제됩니다.

```
┌─────────────────────────────────────────────────┐
│                 Neo4j Cluster                    │
│                                                 │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐    │
│   │ Primary  │  │ Secondary│  │ Secondary│    │
│   │ (Leader) │──│ (Follower)│──│(Follower)│    │
│   │ R/W      │  │ Read     │  │ Read     │    │
│   └──────────┘  └──────────┘  └──────────┘    │
│                                                 │
│   쓰기 → Leader만 처리                          │
│   읽기 → 모든 노드에서 처리 (부하 분산)          │
│   장애 → 자동 리더 선출 (Raft)                  │
└─────────────────────────────────────────────────┘
```

> 💡 **우리 프로젝트는 단일 Docker 인스턴스**입니다.
> 이것은 개발/학습 환경으로 충분합니다.
> 프로덕션으로 전환할 때 클러스터 구성을 검토하세요.

### 17.1.2 백업 전략

**왜 백업이 중요한가:**
온톨로지는 축적된 지식입니다. 데이터 손실은 단순한 파일 유실이 아니라
**수개월의 모델링 작업**을 잃는 것입니다.

**온라인 백업 (neo4j-admin) — Neo4j 실행 중에 수행 가능:**

```bash
# PowerShell에서 실행 (Neo4j가 실행 중인 상태에서 온라인 백업)
# Enterprise Edition에서만 지원. 백업 서비스가 neo4j.conf에서 활성화되어 있어야 함
docker exec neo4j-ontology neo4j-admin database backup \
    --to-path=/backups \
    --type=full \
    neo4j

# 백업 파일 호스트로 복사
docker cp neo4j-ontology:/backups ./backups/
```

**백업 전략 설계:**

| 백업 유형 | 빈도 | 보관 기간 | 용도 |
|-----------|------|-----------|------|
| **풀 백업** | 매일 1회 (새벽) | 30일 | 재해 복구 |
| **증분 백업** | 매 4시간 | 7일 | 빠른 복구 |
| **스냅샷** | 스키마 변경 전 | 영구 | 변경 전 상태 보존 |
| **Cypher 스크립트** | Git에 항상 | 영구 | 스키마 재현 |

> [설계 원칙] **"Cypher 스크립트가 최고의 백업이다"**
>
> 우리 프로젝트에서 `cypher/00_schema.cypher`와 도메인별 Cypher 파일은
> 데이터의 **재현 가능한 원본**입니다. 바이너리 백업이 손상되어도
> Cypher 스크립트만 있으면 처음부터 다시 구축할 수 있습니다.
>
> 기업에서도 마찬가지입니다. ETL 파이프라인과 스키마 스크립트를 Git으로 관리하면,
> "백업에서 복원"이 아니라 "파이프라인을 다시 실행"으로 복구할 수 있습니다.

### 17.1.3 복원

복원은 대상 데이터베이스가 **중지(STOP)** 또는 **삭제(DROP)** 상태여야 합니다.

```cypher
// Neo4j Browser에서: 대상 데이터베이스 중지 (system DB에서 실행)
STOP DATABASE neo4j;
```

```bash
# PowerShell에서: 복원 실행 (컨테이너는 실행 중이어야 함)
docker exec neo4j-ontology neo4j-admin database restore \
    --from-path=/backups \
    --overwrite-destination=true \
    neo4j
```

```cypher
// Neo4j Browser에서: 데이터베이스 다시 시작
START DATABASE neo4j;
```

> ⚠️ **복원 주의사항**
>
> - `--from-path`는 백업 **아티팩트가 있는 디렉토리**를 지정 (특정 파일이 아님)
> - `--overwrite-destination=true`는 기존 데이터를 덮어쓰므로 신중히 사용
> - `docker exec`는 컨테이너가 **실행 중**이어야 동작 (Neo4j 프로세스와 별개)
> - 복원 후 반드시 `START DATABASE`로 데이터베이스를 다시 시작

### 17.1.4 모니터링

**무엇을 모니터링해야 하는가:**

| 지표 | 정상 범위 | 경고 기준 | 확인 방법 |
|------|-----------|-----------|-----------|
| **Heap 사용률** | 50~70% | >85% | JMX 또는 Prometheus |
| **Page Cache 히트율** | >95% | <90% | `dbms.page_cache.hits` |
| **트랜잭션 수** | 서비스에 따라 | 급증/급감 | `db.transaction.count` |
| **쿼리 응답 시간** | <200ms | >1s | Query Log |
| **디스크 사용량** | <70% | >85% | OS 레벨 |
| **GC 빈도** | 낮음 | Full GC 빈번 | JVM GC 로그 |

**Query Log 설정 — 느린 쿼리 감지:**

Neo4j는 설정된 임계값보다 느린 쿼리를 로그에 기록합니다:

```properties
# neo4j.conf (또는 docker-compose.yml의 환경변수)
# 100ms 이상 걸리는 쿼리를 기록
db.logs.query.threshold=100ms

# 쿼리 로그 활성화
db.logs.query.enabled=INFO
```

```bash
# Docker에서 쿼리 로그 확인
docker exec neo4j-ontology cat /var/lib/neo4j/logs/query.log | tail -20
```

**Prometheus + Grafana 연동 (선택):**

Neo4j는 Prometheus 메트릭 엔드포인트를 제공합니다:

```yaml
# docker-compose.yml에 추가
environment:
  - NEO4J_server_metrics_prometheus_enabled=true
  # 포트는 server.metrics.prometheus.endpoint 설정 (기본: localhost:2004)
```

> 💡 Prometheus 엔드포인트가 활성화되면 `http://<neo4j-host>:2004/metrics`에서
> 메트릭을 수집할 수 있습니다. Grafana에서 이 엔드포인트를 데이터 소스로 추가하세요.

## 17.2 스키마 거버넌스

### 17.2.1 왜 거버넌스가 필요한가

> "A팀이 만든 노드 `Server`와 B팀이 만든 노드 `Host`가
> 같은 것인지 다른 것인지 아무도 모릅니다."

개인 프로젝트에서는 스키마를 자유롭게 변경했지만,
기업 환경에서 **여러 팀이 같은 그래프를 사용**하면
규칙 없이는 혼돈이 발생합니다.

### 17.2.2 네이밍 컨벤션

**노드 라벨:**

| 규칙 | 예시 | 이유 |
|------|------|------|
| PascalCase | `ProductCatalog` | Neo4j 관례, 가독성 |
| 단수형 | `Server` (not `Servers`) | 각 노드가 하나의 인스턴스 |
| 도메인 접두사 (대규모) | `Infra_Server`, `HR_Employee` | 팀 간 충돌 방지 |
| 영문 | `Product` (not `제품`) | 코드 호환성, 국제화 |

**관계 타입:**

| 규칙 | 예시 | 이유 |
|------|------|------|
| SCREAMING_SNAKE_CASE | `BELONGS_TO` | Neo4j 관례 |
| 동사 또는 과거분사 | `RUNS_ON`, `CREATED_BY` | 의미 명확 |
| 방향성 내포 | `REPORTS_TO` (부하→상사) | 읽기 쉬움 |

**속성 이름:**

| 규칙 | 예시 | 이유 |
|------|------|------|
| snake_case | `created_at`, `exam_frequency` | 우리 프로젝트 컨벤션 |
| 의미 단위 포함 | `memory_gb` (not `memory`) | 단위 혼동 방지 |
| 불리언 접두사 | `is_active`, `has_children` | 타입 추론 가능 |

**id 속성 규칙 (우리 프로젝트 검증 완료):**

```
✅ 영문 소문자 + 언더스코어 + 숫자: deep_learning, srv_web_01, iso_27001
❌ 대문자, 하이픈, 공백, 한글: Deep-Learning, srv web 01, 딥러닝
```

### 17.2.3 변경 관리 프로세스

```
1. 변경 제안서 작성
   - 추가/수정/삭제할 노드/관계/속성 명세
   - 영향 받는 쿼리/파이프라인 목록
   - 변경 이유와 기대 효과

2. 영향 분석
   - EXPLAIN으로 기존 쿼리 영향 확인
   - 데이터 마이그레이션 필요 여부 판단

3. 리뷰 & 승인
   - 데이터 아키텍트 또는 스키마 오너 승인
   - 코드 리뷰와 동일한 프로세스

4. 마이그레이션 스크립트 작성
   - 멱등성 보장 (IF NOT EXISTS)
   - 롤백 스크립트 함께 작성

5. 스테이징에서 검증
   - 프로덕션 데이터 복사본에서 테스트

6. 프로덕션 적용 + 모니터링
   - 점검 시간에 적용, 쿼리 성능 모니터링
```

**스키마 변경 Cypher 템플릿:**

```cypher
// ================================================================
// 스키마 변경: v2.1 → v2.2
// 변경 내용: Product 노드에 category_path 속성 추가
// 변경 일자: 2025-03-15
// 변경자: 데이터팀
// 영향: product_search 쿼리, ETL 파이프라인
// 롤백: 아래 롤백 섹션 참조
// ================================================================

// --- 변경 적용 ---

// 1. 새 인덱스 추가 (IF NOT EXISTS로 멱등성 보장)
CREATE INDEX product_category_path IF NOT EXISTS
FOR (p:Product) ON (p.category_path);

// 2. 기존 데이터 마이그레이션
MATCH (p:Product)-[:BELONGS_TO]->(c:Category)
SET p.category_path = c.id;

// --- 검증 ---
// MATCH (p:Product) WHERE p.category_path IS NULL RETURN count(p);
// 결과가 0이면 성공

// --- 롤백 (필요 시) ---
// MATCH (p:Product) REMOVE p.category_path;
// DROP INDEX product_category_path IF EXISTS;
```

### 17.2.4 스키마 문서화

**스키마 레지스트리 — 노드/관계/속성 카탈로그:**

| 항목 | 내용 | 관리 위치 |
|------|------|-----------|
| 노드 라벨 카탈로그 | 라벨명, 목적, 필수 속성, 오너 | Wiki 또는 Confluence |
| 관계 타입 카탈로그 | 타입명, 방향, 출발/도착 라벨, 속성 | Wiki |

> 💡 **apoc.meta.graph()가 라이브 문서다**
>
> 코드와 문서가 따로 관리되면 결국 불일치합니다.
> `CALL apoc.meta.graph()`는 **현재 그래프의 실제 구조**를 보여줍니다.
> 이것을 "살아있는 스키마 문서"로 활용하고,
> Wiki에는 "왜 이렇게 설계했는지"만 기록하세요.

### 17.2.5 외부 표준과의 정렬 (SKOS / Dublin Core)

기업 온톨로지가 다른 시스템과 데이터를 교환하거나,
산업 표준에 맞춰야 할 때는 **외부 어휘(Vocabulary)와의 정렬(Alignment)**이 필요합니다.

**왜 정렬이 필요한가:**

| 상황 | 문제 | 정렬의 효과 |
|------|------|------------|
| 부서 A와 부서 B가 각자 온톨로지를 구축 | "Category"와 "Classification"이 같은 것인지 모름 | SKOS:broader/narrower로 표준화 |
| 외부 파트너에게 그래프 데이터를 제공 | 우리만의 관계명을 이해 못함 | Dublin Core 속성으로 매핑 |
| 규제 기관이 RDF 형식을 요구 | Property Graph를 직접 전달 불가 | n10s로 RDF 변환 후 전달 |

**우리 스키마와 SKOS/Dublin Core 대응:**

| 우리 스키마 | SKOS 대응 | Dublin Core 대응 |
|------------|-----------|------------------|
| `SUBCLASS_OF` | `skos:broader` (상위 개념) | — |
| `HAS_TYPE` | `skos:narrower` (하위 개념) | — |
| `name_kr` | `skos:prefLabel@ko` (선호 라벨) | `dc:title` |
| `name` | `skos:prefLabel@en` | `dc:title` |
| `definition` | `skos:definition` | `dc:description` |
| `aliases` | `skos:altLabel` (대안 라벨) | — |
| `COMPARED_WITH` | `skos:related` (관련 개념) | — |
| `domain` | `skos:inScheme` (소속 체계) | `dc:subject` |
| `MENTIONED_IN` → Document | — | `dc:source` (출처) |

**n10s로 매핑 설정 (Neo4j Browser에서 실행):**

```cypher
// n10s 매핑: 우리의 속성명 → SKOS/Dublin Core 어휘
CALL n10s.nsprefixes.add('skos', 'http://www.w3.org/2004/02/skos/core#');
CALL n10s.nsprefixes.add('dc', 'http://purl.org/dc/elements/1.1/');

// 속성 매핑
CALL n10s.mapping.add('http://www.w3.org/2004/02/skos/core#prefLabel', 'name_kr');
CALL n10s.mapping.add('http://www.w3.org/2004/02/skos/core#definition', 'definition');
CALL n10s.mapping.add('http://www.w3.org/2004/02/skos/core#altLabel', 'aliases');

// 관계 매핑
CALL n10s.mapping.add('http://www.w3.org/2004/02/skos/core#broader', 'SUBCLASS_OF');
CALL n10s.mapping.add('http://www.w3.org/2004/02/skos/core#related', 'COMPARED_WITH');
```

> 💡 **"지금 당장 필요하지 않더라도 매핑 가능성을 열어두라"**
>
> n10s 매핑을 설정해두면, 나중에 `CALL n10s.rdf.export.cypher(...)` 한 줄로
> Property Graph를 RDF/SKOS 형식으로 내보낼 수 있습니다.
> 우리 프로젝트에 이미 n10s가 설치되어 있으므로, 추가 비용 없이 활용 가능합니다.
>
> **SKOS 개요:**
> - **SKOS**(Simple Knowledge Organization System)는 W3C 표준으로,
>   분류 체계/시소러스/택소노미를 RDF로 표현하는 어휘입니다.
> - `skos:broader` = 상위 개념, `skos:narrower` = 하위 개념,
>   `skos:related` = 관련 개념, `skos:prefLabel` = 선호 라벨
>
> **Dublin Core 개요:**
> - **Dublin Core**는 문서/리소스의 메타데이터 표준 (15개 핵심 요소)
> - `dc:title`, `dc:description`, `dc:subject`, `dc:source` 등

### 17.3.4 애플리케이션 연동 — Neo4j Python Driver

기업 환경에서 Cypher를 실행하는 주된 방법은 Neo4j Browser가 아니라
**애플리케이션 코드(Python, JavaScript 등)에서 Driver를 통해 호출**하는 것입니다.

**기본 연결 패턴:**

```python
# pip install neo4j
from neo4j import GraphDatabase

driver = GraphDatabase.driver(
    "bolt://localhost:7687",
    auth=("neo4j", "ontology2025!")
)

# 읽기 쿼리
def get_concepts_by_domain(domain_code):
    with driver.session() as session:
        result = session.run(
            "MATCH (c:Concept {domain: $domain}) "
            "RETURN c.id, c.name_kr, c.definition "
            "ORDER BY c.name_kr",
            domain=domain_code  # 파라미터 바인딩 (SQL Injection 방지)
        )
        return [record.data() for record in result]

# 쓰기 쿼리 (트랜잭션 단위)
def merge_concept(concept_id, name_kr, domain):
    with driver.session() as session:
        session.execute_write(
            lambda tx: tx.run(
                "MERGE (c:Concept {id: $id}) "
                "SET c.name_kr = $name_kr, c.domain = $domain",
                id=concept_id, name_kr=name_kr, domain=domain
            )
        )

# 사용 예시
concepts = get_concepts_by_domain("AI")
merge_concept("transfer_learning", "전이학습", "AI")

# 종료
driver.close()
```

**핵심 규칙:**

| 규칙 | 이유 |
|------|------|
| `$parameter` 바인딩 사용 | SQL Injection 방지, 쿼리 캐싱 |
| 읽기는 `session.run()` | 자동 트랜잭션 |
| 쓰기는 `session.execute_write()` | 명시적 쓰기 트랜잭션, 자동 재시도 |
| `driver.close()` 호출 | 연결 풀 해제 |

> ⚠️ **문자열 포맷팅으로 Cypher를 조립하지 마세요**
>
> ```python
> # ❌ 위험! Cypher Injection 가능
> session.run(f"MATCH (c:Concept {{id: '{user_input}'}}) RETURN c")
>
> # ✅ 안전! 파라미터 바인딩
> session.run("MATCH (c:Concept {id: $id}) RETURN c", id=user_input)
> ```

```cypher
GRANT NAME MANAGEMENT ON DATABASE * TO schema_admin;
```

## 17.5 성능 운영

### 17.5.1 메모리 설정 — 가장 중요한 성능 요소

Neo4j의 성능은 **메모리 설정**에 가장 크게 좌우됩니다.

```
┌─────────────────────────────────────────────────┐
│              서버 물리 메모리                     │
│                                                 │
│  ┌───────────┐  ┌───────────────┐  ┌─────────┐ │
│  │  JVM Heap │  │  Page Cache   │  │   OS    │ │
│  │  (쿼리,   │  │  (디스크→메모리│  │ + 기타  │ │
│  │   트랜잭션)│  │   캐시)       │  │         │ │
│  │  ~4~8GB   │  │  가능한 최대  │  │  ~2GB   │ │
│  └───────────┘  └───────────────┘  └─────────┘ │
└─────────────────────────────────────────────────┘
```

| 메모리 영역 | 역할 | 설정 | 권장값 |
|-------------|------|------|--------|
| **Heap** | 쿼리 실행, 트랜잭션 처리 | `server.memory.heap.initial_size` / `max_size` | 4~8GB |
| **Page Cache** | 그래프 데이터 캐싱 | `server.memory.pagecache.size` | 데이터 크기의 120%+ |
| **OS** | 파일 I/O, 시스템 | (자동) | 최소 2GB |

**설정 예시 (Docker):**

```yaml
# docker-compose.yml
environment:
  # 32GB 서버 기준
  - NEO4J_server_memory_heap_initial__size=4G
  - NEO4J_server_memory_heap_max__size=4G
  - NEO4J_server_memory_pagecache_size=20G
  # Transaction 메모리 제한 (대규모 쿼리 OOM 방지)
  - NEO4J_db_memory_transaction_total_max=4G
```

> 💡 **Page Cache가 데이터보다 커야 하는 이유**
>
> Page Cache가 전체 데이터를 담을 수 있으면,
> 모든 쿼리가 디스크 I/O 없이 메모리에서 처리됩니다.
> 이것만으로 쿼리 속도가 **10~100배** 빨라질 수 있습니다.
>
> 데이터 크기 확인:
> ```bash
> docker exec neo4j-ontology du -sh /data/databases/neo4j/
> ```

### 17.5.4 성장 예측과 용량 계획


**용량 계획 공식 (대략적):**

```

메모리 = 디스크 크기 × 1.2 (Page Cache) + 4~8GB (Heap) + 2GB (OS)

예시: 1,000만 노드 + 5,000만 관계
디스크 ≈ (10M × 15) + (50M × 34) + 속성 ≈ 약 3~5GB
메모리 ≈ 5GB × 1.2 + 6GB + 2GB ≈ 약 14GB
→ 16~32GB 서버 권장
```

### 17.5.6 동시성 관리

기업 환경에서는 **여러 ETL 파이프라인이 동시에** 같은 그래프에 데이터를 적재합니다.
이때 발생할 수 있는 문제와 대응 방법을 다룹니다.

**문제 1: MERGE 경쟁 조건 (Race Condition)**

두 트랜잭션이 **동시에** 같은 id로 MERGE를 실행하면:

```
트랜잭션 A: MERGE (c:Concept {id: 'new_concept'}) SET c.name = 'A'
트랜잭션 B: MERGE (c:Concept {id: 'new_concept'}) SET c.name = 'B'
```

UNIQUE 제약조건이 있으면 하나는 성공, 하나는 `ConstraintValidationFailed`로 실패합니다.
이것은 **의도된 동작**이지만, 실패한 트랜잭션을 재시도해야 합니다.

**대응 — Python Driver 재시도 패턴:**

```python
from neo4j import GraphDatabase
from neo4j.exceptions import TransientError

def merge_with_retry(session, concept_id, name_kr, domain, max_retries=3):
    for attempt in range(max_retries):
        try:
            session.execute_write(
                lambda tx: tx.run(
                    "MERGE (c:Concept {id: $id}) "
                    "SET c.name_kr = $name_kr, c.domain = $domain",
                    id=concept_id, name_kr=name_kr, domain=domain
                )
            )
            return  # 성공
        except TransientError:
            if attempt == max_retries - 1:
                raise  # 최종 실패
            # 재시도 (Neo4j Driver는 execute_write에서 자동 재시도를 지원)
```

> 💡 `session.execute_write()`는 `TransientError` 발생 시 **자동으로 재시도**합니다.
> 별도 재시도 로직을 작성하지 않아도 대부분의 경쟁 조건이 해결됩니다.

**문제 2: 데드락 (Deadlock)**

두 트랜잭션이 서로의 노드를 잠그면 교착 상태가 발생합니다.
Neo4j는 데드락을 감지하면 한 트랜잭션을 **자동으로 롤백**합니다.

**대응:** 동일 노드를 수정하는 배치 작업은 **순차 실행**하거나,
`CALL IN TRANSACTIONS`로 배치 크기를 제어하세요.

**동시성 관리 규칙:**

| 규칙 | 이유 |
|------|------|
| UNIQUE 제약조건 필수 | MERGE 경쟁 시 중복 방지의 마지막 안전장치 |
| 쓰기는 `execute_write()` | 자동 재시도 + 올바른 트랜잭션 라우팅 |
| 대량 적재는 순차 실행 | 데드락 위험 최소화 |
| 배치 크기 500~1,000 | 트랜잭션 잠금 범위 제한 |

## 17.6 Change Data Capture (CDC) — 실시간 변경 추적

프로덕션 그래프에서 "누가, 언제, 무엇을 변경했는가?"를 추적하는 것은
거버넌스의 핵심입니다. Neo4j의 **CDC(Change Data Capture)**는 노드/관계의
생성, 수정, 삭제를 실시간으로 캡처하여 다른 시스템과 동기화할 수 있게 합니다.

### 17.6.1 CDC란 무엇인가

```
┌──────────┐    변경 발생     ┌──────────┐    CDC 캡처    ┌────────────────┐
│ Neo4j DB │ ──────────────→ │ TX Log   │ ────────────→ │ 외부 시스템     │
│          │  (MERGE, SET,   │ (WAL)    │               │ (Kafka, ES,    │
│          │   DELETE)       │          │               │  Data Lake)    │
└──────────┘                 └──────────┘               └────────────────┘
```

CDC는 Neo4j의 트랜잭션 로그(WAL)를 기반으로 동작하며,
변경된 엔티티의 **이전 상태와 이후 상태**를 모두 제공합니다.

### 17.6.2 CDC 활성화 (Neo4j 5.13+)

```
# neo4j.conf 또는 docker-compose.yml 환경변수
NEO4J_db_tx__log_rotation_retention__policy=2 days
```

### 17.6.3 CDC 쿼리 — 변경 사항 조회

```cypher
// 1. 현재 변경 ID 획득 (시작점 설정)
CALL db.cdc.current() YIELD id AS currentChangeId
RETURN currentChangeId;
// → 이 ID를 저장해두면, 이후 이 ID 이후의 변경만 조회 가능

// 2. 특정 시점 이후의 모든 변경 조회
CALL db.cdc.query($savedChangeId)
YIELD txId, changeType, entityType, properties, changedLabels
RETURN txId, changeType, entityType, properties
ORDER BY txId;
// changeType: 'n' (노드) 또는 'r' (관계)
// entityType: 'CREATE', 'UPDATE', 'DELETE'

// 3. Concept 노드 변경만 필터링
CALL db.cdc.query($savedChangeId, [{
    select: "n",
    labels: ["Concept"]
}])
YIELD txId, entityType, properties
RETURN txId, entityType, properties;

// 4. SUBCLASS_OF 관계 변경만 필터링
CALL db.cdc.query($savedChangeId, [{
    select: "r",
    type: "SUBCLASS_OF"
}])
YIELD txId, entityType, properties
RETURN txId, entityType, properties;
```

### 17.6.4 CDC 활용 시나리오

| 시나리오 | CDC 활용 방법 | 비고 |
|----------|--------------|------|
| **감사 로그** | 변경 이력을 별도 DB에 적재 | 규제 준수 (GDPR, ISO 27001) |
| **외부 동기화** | Kafka로 스트리밍 → Elasticsearch 인덱싱 | 검색 엔진 실시간 업데이트 |
| **캐시 무효화** | 변경된 노드의 캐시 자동 제거 | API 서버 캐시 일관성 |
| **알림** | 특정 라벨/관계 변경 시 Slack 알림 | 스키마 거버넌스 |
| **RDB 동기화** | 그래프 변경 → RDB 테이블 업데이트 | 병행 운영 시 |

### 17.6.5 우리 프로젝트에서의 적용

현재 개발 단계에서는 CDC가 필수는 아니지만, **STEP 7(전체 적재) 이후** 다음과 같이 활용할 수 있습니다:

```cypher
// 예: "오늘 추가된 Concept 노드 확인" — 적재 검증 자동화
CALL db.cdc.query($yesterdayChangeId, [{
    select: "n",
    labels: ["Concept"],
    operation: "c"   // CREATE만
}])
YIELD properties
RETURN properties.after.id AS new_concept,
       properties.after.name_kr AS name
ORDER BY new_concept;
```

> 💡 **CDC vs 수동 감사 로그**
>
> 수동 감사: `SET c.modified_at = datetime()` — 직접 관리해야 하며, 삭제는 추적 불가
> CDC: Neo4j가 자동으로 모든 변경을 캡처 — 삭제, 관계 변경까지 포함
> 프로덕션에서는 CDC를 기본으로 활성화하고, 수동 감사는 보조 수단으로 사용하세요.
