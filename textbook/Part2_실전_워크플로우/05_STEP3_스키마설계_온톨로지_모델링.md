## 5.1 온톨로지란 무엇인가

### 5.1.1 철학에서 정보과학으로

"온톨로지(Ontology)"라는 단어를 처음 들으면 철학 용어처럼 느껴질 수 있습니다.
실제로 그 기원은 철학에 있습니다.

**철학적 온톨로지**는 고대 그리스 아리스토텔레스에서 시작됩니다.
그는 "존재하는 것들을 분류하고 범주화"하려 했습니다.
"이 세상에는 어떤 종류의 것들이 존재하며, 그것들은 어떤 관계를 맺고 있는가?"
이 질문이 온톨로지의 출발점입니다.

이 철학적 질문은 1990년대 인공지능 연구에서 **정보과학적 온톨로지**로 재탄생합니다.
AI 시스템이 세계를 "이해"하려면, 세계의 구조를 명시적으로 표현할 수 있어야 했기 때문입니다.

1993년, 스탠퍼드 대학의 Tom Gruber는 정보과학 온톨로지를 이렇게 정의했습니다:

> **"An ontology is an explicit specification of a shared conceptualization."**
> (온톨로지란 공유된 개념화의 명시적 명세이다.)

이 한 줄에 온톨로지의 핵심이 담겨 있습니다. 하나씩 뜯어보겠습니다:

| 용어 | 의미 | 우리 프로젝트에서 |
|------|------|-------------------|
| **Conceptualization** (개념화) | 세상의 일부를 추상적으로 모델링한 것 | IT 도메인의 개념, 기술, 방법론을 모델링 |
| **Shared** (공유된) | 여러 사람/시스템이 합의한 모델 | 기술사 시험이라는 공통 맥락에서의 합의 |
| **Explicit** (명시적) | 암묵적이 아닌, 문서화되고 형식화된 | Neo4j 그래프로 구조화 |
| **Specification** (명세) | 구체적인 규칙과 제약조건 포함 | 노드 타입, 관계 타입, 속성 규칙 |

즉, 온톨로지는 **"특정 영역의 지식을 구조화하여 명시적으로 표현한 것"**입니다.

> 💡 **쉽게 말하면**: 온톨로지는 "이 분야에서 중요한 것들은 무엇이고,
> 그것들이 서로 어떤 관계를 맺고 있는지를 정리한 지도"입니다.

### 5.1.2 온톨로지의 4계층

온톨로지는 그 형식성(formality)의 수준에 따라 4개 계층으로 나뉩니다.
이 계층 분류는 Deborah McGuinness(2003)의 "Ontology 101" 논문과
Guarino(1998)의 형식 온톨로지 이론에 기반합니다.
아래로 갈수록 단순하고, 위로 갈수록 엄격합니다.

```
┌──────────────────────────────────────────────────────────────┐
│ Level 4: 형식 온톨로지 (Formal Ontology)                      │
│          OWL/RDF 기반, 논리적 추론 가능                        │
│          "A가 B의 하위 클래스이고 B가 C의 하위이면,             │
│           A는 자동으로 C의 하위이다"                            │
│          예: 시맨틱웹, 생의학 온톨로지 (SNOMED CT, Gene Ontology)│
├──────────────────────────────────────────────────────────────┤
│ Level 3: 도메인 온톨로지 (Domain Ontology)  ← 우리가 여기!     │
│          특정 도메인의 개념/관계를 구조화                       │
│          논리적 추론은 없지만, 풍부한 의미 관계                  │
│          예: IT 지식그래프, 기업 제품 카탈로그                   │
├──────────────────────────────────────────────────────────────┤
│ Level 2: 택소노미 (Taxonomy)                                  │
│          계층적 분류 체계                                      │
│          "A는 B의 한 종류이다" 관계만 존재                     │
│          예: 생물 분류 (계-문-강-목-과-속-종), 도서 분류       │
├──────────────────────────────────────────────────────────────┤
│ Level 1: 용어집/시소러스 (Glossary/Thesaurus)                 │
│          용어 정의 + 동의어/유의어 관계                        │
│          예: 검색엔진 동의어 사전, 의학 용어집                 │
└──────────────────────────────────────────────────────────────┘
```

**우리 프로젝트는 Level 3 — 도메인 온톨로지**를 구축합니다.

왜 Level 4(형식 온톨로지)가 아닌가?
- Level 4는 OWL/RDF를 사용하여 논리적 추론이 가능하지만, 구축 비용이 높고 쿼리 유연성이 낮습니다
- Level 3은 Property Graph(Neo4j)를 사용하여 유연하고 빠르게 구축/수정할 수 있습니다
- 우리의 목적(지식 검색, 관계 탐색, GraphRAG)에는 Level 3이 최적입니다

왜 Level 2(택소노미)로 충분하지 않은가?
- 택소노미는 "상위-하위" 관계만 표현합니다
- 우리는 "비교(COMPARED_WITH)", "위협(THREATENS)", "구현(IMPLEMENTS)" 등 다양한 관계가 필요합니다
- 기출문제의 57.4%가 단답설명형(정의+관계), 11.3%가 비교형 → 풍부한 관계 표현 필수

### 5.1.3 온톨로지 vs 택소노미 vs 시소러스 vs 스키마

이 용어들은 자주 혼동됩니다. 명확히 구분해두겠습니다.

| 구분 | 용어집/시소러스 | 택소노미 | 도메인 온톨로지 | DB 스키마 |
|------|----------------|----------|----------------|-----------|
| **핵심 표현** | 용어 + 동의어 | 계층 분류 | 개념 + 다양한 관계 | 테이블 + 외래키 |
| **관계 유형** | 동의어, 유사어 | IS-A (상위-하위) | IS-A, HAS-A, 비교, 의존 등 다수 | FK, JOIN |
| **구조** | 평면적 (flat) | 트리 (tree) | 그래프 (graph) | 테이블 (table) |
| **추론** | 없음 | 단순 상속 | 경로 탐색 기반 | SQL 기반 |
| **유연성** | 매우 높음 | 보통 | 높음 | 낮음 (스키마 변경 비용) |
| **적합한 도구** | 사전, 검색엔진 | 파일 시스템, 카테고리 | Neo4j, OWL/RDF | MySQL, PostgreSQL |
| **예시** | "암호화 = Encryption = 暗号化" | 동물-포유류-고양이 | 딥러닝→머신러닝→AI (+ 여러 관계) | users 테이블 |

> 💡 **핵심 차이**: 택소노미는 "트리(나무)" 구조, 온톨로지는 "그래프(그물)" 구조입니다.
> 현실 세계의 지식은 트리보다 그물에 가깝습니다.
> "딥러닝"은 "머신러닝"의 하위이면서, "GPU"에 의존하고, "이미지 분류"에 사용되고,
> "CNN"이 구현하며, "적대적 공격"에 위협받습니다.
> 이런 복잡한 관계는 트리로 표현할 수 없습니다.

### 5.1.4 Property Graph에서의 온톨로지

온톨로지를 구현하는 기술은 크게 두 갈래로 나뉩니다:

**갈래 1: RDF/OWL (시맨틱웹)**
```
@prefix : <http://example.org/it#> .
:DeepLearning rdf:type :Concept ;
    rdfs:subClassOf :MachineLearning ;
    :dependsOn :GPU .
```
- W3C 국제 표준
- 논리적 추론 엔진 (OWL Reasoner)이 새로운 사실을 자동 도출
- 단점: 학습 곡선 높음, 쿼리 언어(SPARQL) 복잡, 속성 표현 제한적

**갈래 2: Property Graph (Neo4j)** ← 우리의 선택
```cypher
MERGE (dl:Concept {id: 'deep_learning'})
SET dl.name = 'Deep Learning', dl.name_kr = '딥러닝'

MERGE (ml:Concept {id: 'machine_learning'})
MERGE (dl)-[:SUBCLASS_OF]->(ml)
```
- 직관적인 노드-관계 모델
- Cypher 쿼리 언어가 읽기 쉬움
- 노드/관계 모두에 풍부한 속성(Property) 부여 가능
- 단점: 표준화 부족, 자동 추론 없음 (수동으로 관계 생성 필요)

> ⚠️ **Neo4j 공식 네이밍 컨벤션 참고**
>
> Neo4j 공식 문서는 다음을 권장합니다:
> - 노드 라벨: **PascalCase** (예: `:VehicleOwner`)
> - 관계 타입: **SCREAMING_SNAKE_CASE** (예: `:OWNS_VEHICLE`)
> - 속성 키: **camelCase** (예: `firstName`)
>
> 우리 프로젝트는 속성에 **snake_case**를 사용합니다 (예: `name_kr`, `exam_frequency`).
> 이는 의도적 선택입니다: Python 파이프라인과의 일관성, 가독성 우선.
> 공식 권장과 다르지만 프로젝트 내 일관성이 유지되면 문제없습니다.
> **중요한 것은 "프로젝트 내 일관성"입니다. 혼용만 피하면 됩니다.**

**선택 기준 정리:**

| 기준 | RDF/OWL | Property Graph (Neo4j) |
|------|---------|----------------------|
| 구축 속도 | 느림 | **빠름** |
| 스키마 변경 | 어려움 | **매우 쉬움** (스키마리스) |
| 쿼리 유연성 | 보통 | **높음** (Cypher) |
| 자동 추론 | **있음** (OWL Reasoner) | 없음 (수동) |
| 표준 교환 | **W3C 표준** | 비표준 |
| 속성 표현 | 제한적 (reification 필요) | **풍부** (노드/관계 속성) |
| 학습 곡선 | 높음 | **낮음** |
| 시각화 | 제한적 | **우수** (Neo4j Browser, Bloom) |

우리 프로젝트에서 Property Graph를 선택한 이유:

1. **"첫 스키마는 틀린다"** — 빠르게 만들고 빠르게 수정해야 하므로 스키마리스가 유리
2. **풍부한 속성 필요** — definition, exam_frequency, perspectives 등 속성이 많음
3. **구축 속도** — 2,392개 문서를 처리해야 하므로 빠른 프로토타이핑 필수
4. **쿼리 유연성** — "3-hop 이내 연결된 모든 개념" 같은 탐색이 핵심 사용 패턴
5. **시각화** — Neo4j Browser에서 즉시 그래프를 눈으로 확인 가능

> **[기업 박스] RDF가 더 적합한 경우**
>
> 다음 조건에 해당하면 Property Graph 대신 RDF/OWL을 고려하세요:
> - 산업 표준 온톨로지(SNOMED CT, FIBO 등)와 연동해야 하는 경우
> - 여러 조직 간 온톨로지를 교환/통합해야 하는 경우
> - 자동 추론이 핵심 요구사항인 경우 (예: 의료 진단 시스템)
> - 규제 기관이 RDF/OWL 형식을 요구하는 경우
>
> Neo4j도 n10s 플러그인을 통해 RDF/OWL을 임포트할 수 있으므로,
> "Property Graph로 시작 → 필요 시 RDF 연동"이라는 점진적 전략도 가능합니다.

### 5.1.5 상위 온톨로지(Upper Ontology)와 우리 스키마의 관계

온톨로지 공학에는 **상위 온톨로지**라는 개념이 있습니다.
도메인에 독립적인 "존재의 최상위 분류 체계"로,
모든 도메인 온톨로지의 기초 뼈대 역할을 합니다.

대표적인 상위 온톨로지:

| 상위 온톨로지 | 핵심 구분 | 설명 |
|-------------|----------|------|
| **BFO** (Basic Formal Ontology) | Continuant vs Occurrent | 지속하는 것(물체, 개념) vs 일어나는 것(과정, 이벤트) |
| **DOLCE** | Endurant vs Perdurant | BFO와 유사, 유럽 학계 선호 |
| **SUMO** | Physical vs Abstract | 물리적 존재 vs 추상적 존재 |

**우리 스키마와의 대응:**

```
BFO의 Continuant (지속하는 것)
  ├── Concept     ← "딥러닝"은 시간이 지나도 딥러닝
  ├── Technology  ← "Docker"는 지속하는 도구
  ├── Standard    ← "ISO 27001"은 지속하는 규격
  └── Domain      ← "AI 분야"는 지속하는 범주

BFO의 Occurrent (일어나는 것)
  ├── Method      ← "K-Means"는 절차(과정)
  └── ExamSession ← "132회 시험"은 일어난 이벤트
```

이 대응을 의식적으로 만든 것은 아니지만,
결과적으로 우리의 Concept/Technology(Continuant)와 Method(Occurrent) 구분은
**상위 온톨로지의 기본 구분과 일치**합니다.

> 💡 **이것을 알아야 하는 이유**
>
> "Concept과 Method를 왜 나누는가?"라는 질문에
> "직관적으로 다르니까"가 아니라
> **"존재론적으로 '지속하는 것'과 '일어나는 것'이 근본적으로 다르기 때문"**이라고
> 답할 수 있습니다. 이것이 노드 타입 분리의 철학적 근거입니다.
>
> 실무에서 상위 온톨로지를 직접 구현할 필요는 없지만,
> 설계 판단이 모호할 때 "이것은 Continuant(지속)인가 Occurrent(과정)인가?"를
> 기준으로 삼으면 일관성이 높아집니다.

### 5.1.6 온톨로지 설계 패턴 (ODP)

소프트웨어 공학에 디자인 패턴(GoF)이 있듯이,
온톨로지 공학에도 **Ontology Design Patterns (ODP)**가 있습니다.
반복적으로 등장하는 모델링 문제에 대한 검증된 해법을 패턴으로 정리한 것입니다.

**우리 프로젝트에서 사용 중인 ODP:**

| ODP 패턴 | 우리의 구현 | 해결하는 문제 |
|----------|-----------|--------------|
| **Taxonomy Pattern** | `SUBCLASS_OF` 체인 | 개념의 계층적 분류 |
| **Part-Whole Pattern** | `HAS_COMPONENT` | 전체-부분 관계 (CIA→기밀성) |
| **Participation Pattern** | `IMPLEMENTS` | 엔티티가 다른 엔티티에 참여 (SVM→지도학습) |
| **Classification Pattern** | `BELONGS_TO` → Domain | 인스턴스의 카테고리 분류 |
| **Provenance Pattern** | `MENTIONED_IN` → Document | 데이터의 출처 추적 |
| **Comparison Pattern** | `COMPARED_WITH` + perspectives | 두 개체의 다차원 비교 |

> 💡 **ODP를 알아야 하는 이유**
>
> 스키마를 "직관적으로" 설계하면 개인 판단에 의존하게 됩니다.
> ODP를 참조하면 **"이 문제는 Taxonomy Pattern으로 해결한다"**처럼
> 검증된 해법을 적용할 수 있습니다.
>
> **심화 학습**: *Ontology Engineering with Ontology Design Patterns* (IOS Press)
> 또는 [ontologydesignpatterns.org](http://ontologydesignpatterns.org/)를 참조하세요.

## 5.2 Question-Driven Modeling (QDM) 방법론

### 5.2.1 핵심 철학

> **QDM의 학술적 맥락**
>
> "Question-Driven Modeling"은 이 교재에서 사용하는 명칭이며,
> 그래프 데이터 모델링 커뮤니티에서 널리 공유되는 접근법입니다.
> Neo4j의 Jesús Barrasa는 "Going Meta" 시리즈에서 "질문에서 시작하는 모델링"을 강조했고,
> Neo4j 공식 문서(Getting Started > Data Modeling)도
> "비즈니스 요구사항(질문)이 모델을 결정한다"고 안내합니다.
> 또한 Jim Webber(Neo4j Chief Scientist)는
> "모델은 당신이 데이터에 던질 질문에 의해 형성되어야 한다"고 반복 강조합니다.
> 이 교재의 QDM 5단계는 이러한 원칙을 체계적 프로세스로 정리한 것입니다.

스키마 설계에서 가장 흔한 실수는 **"데이터를 먼저 보고 구조를 만드는 것"**입니다.
데이터에는 구조가 없습니다. 구조는 **질문**에서 나옵니다.

> 📌 **[설계 원칙] "질문이 스키마를 결정한다"**
>
> 좋은 스키마란 무엇인가? 예쁜 다이어그램? 정규화된 구조?
> 아닙니다. **좋은 스키마 = 답변해야 할 질문에 효율적으로 답할 수 있는 구조**입니다.

이것이 **Question-Driven Modeling (QDM)**의 핵심 철학입니다.

스키마를 설계할 때, 이것을 먼저 물어야 합니다:

```
"이 지식그래프에 어떤 질문을 던질 것인가?"
```

질문이 결정되면, 각 질문에 답하기 위해 필요한 데이터 구조가 자연스럽게 드러납니다.

**예시로 바로 이해해봅시다:**

질문: *"딥러닝의 하위 기술에는 무엇이 있는가?"*

이 질문에 답하려면:
- "딥러닝"이라는 개념이 존재해야 합니다 → **Concept 노드** 필요
- "하위 기술"이라는 관계가 표현되어야 합니다 → **SUBCLASS_OF 관계** 필요
- CNN, RNN, GAN 등이 딥러닝의 하위로 연결되어야 합니다

```cypher
// 이 질문에 답하는 Cypher
MATCH (sub)-[:SUBCLASS_OF]->(dl:Concept {id: 'deep_learning'})
RETURN sub.name_kr AS 하위기술
```

질문 하나에서 노드 타입 1개(Concept)와 관계 타입 1개(SUBCLASS_OF)가 도출되었습니다.
이것이 QDM의 힘입니다.

> 📌 **[설계 원칙] "답변할 수 없는 질문이 있다면, 스키마가 부족한 것이다"**
>
> 스키마를 설계한 후 이렇게 점검하세요:
> "우리가 중요하다고 정의한 질문들에 모두 답할 수 있는가?"
> 답할 수 없는 질문이 있다면 → 노드, 관계, 또는 속성이 빠진 것입니다.

### 5.2.2 QDM 5단계 프로세스

QDM은 5단계의 체계적 프로세스로 구성됩니다.

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  ① 핵심 질문 도출                                       │
│      ↓                                                  │
│  ② 답변 패턴 분석                                       │
│      ↓                                                  │
│  ③ 노드 후보 식별                                       │
│      ↓                                                  │
│  ④ 관계 패턴 도출                                       │
│      ↓                                                  │
│  ⑤ 속성 확정                                            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

각 단계를 상세히 살펴보겠습니다.

---

#### ① 핵심 질문 도출

스키마 설계의 출발점은 **"이 시스템에 어떤 질문을 할 것인가?"**를 정의하는 것입니다.

질문의 출처는 다양합니다:

| 출처 | 방법 | 우리 프로젝트의 경우 |
|------|------|---------------------|
| 이해관계자 인터뷰 | "무엇이 궁금하세요?" | 기술사 시험 응시자: "이 개념과 관련된 것은?" |
| 기존 보고서/쿼리 | 어떤 보고서를 만들고 있는지 분석 | 기출문제 22,518줄이 곧 "질문" 목록 |
| 비즈니스 프로세스 | 업무 흐름에서 질문 추출 | 기술사 답안 작성 프로세스 |
| 데이터 정량 분석 | 빈도, 분포에서 중요 질문 식별 | 문제 유형 분포 → 질문 패턴 |

**우리 프로젝트는 특별한 이점이 있습니다:**
기출문제 9,291건 자체가 이미 "핵심 질문"입니다.
문제 유형 분석이 곧 질문 분류입니다.

```
기출문제 유형 분포 (= 질문 유형 분포):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
단답설명형  ████████████████████████████  57.4%  "X를 설명하시오"
서술형(3항목) ██████████              20.2%  "가.개념 나.구성요소 다.절차"
비교형       █████                    11.3%  "A와 B를 비교하시오"
코드형       ██                        4.4%  "코드를 작성하시오"
계산형       █                         3.0%  "~를 계산하시오"
기타         ██                        3.7%  서술형(2,4항목), 다이어그램
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

이 분포에서 바로 핵심 질문 유형이 도출됩니다:

1. **정의 질문** (57.4%): "X란 무엇인가? 정의하고 설명하시오."
2. **구조 질문** (20.2%): "X의 구성요소/절차/단계를 설명하시오."
3. **비교 질문** (11.3%): "A와 B를 비교하시오."
4. **관계 질문**: "X는 어떤 기술을 사용하는가?" "Y의 상위 개념은?"
5. **분류 질문**: "보안 도메인의 핵심 개념 Top 10은?"

---

#### ② 답변 패턴 분석

각 질문 유형이 요구하는 **답변의 구조**를 분석합니다.
답변 구조가 곧 데이터 구조를 결정합니다.

**패턴 1: 정의 질문 → "개체 + 정의"**
```
Q: "딥러닝을 설명하시오."
A: "딥러닝(Deep Learning)이란 인공신경망을 여러 층으로 쌓아
    복잡한 패턴을 학습하는 머신러닝의 한 분야이다."

필요한 데이터 구조:
- 개체(딥러닝)가 존재해야 함 → 노드
- 정의가 저장되어야 함 → 속성 (definition)
- 상위 개념(머신러닝)과의 관계 → 관계 (SUBCLASS_OF)
```

**패턴 2: 구조 질문 → "개체 + 구성요소 목록"**
```
Q: "정보보안의 3대 요소를 설명하시오."
A: "가. 기밀성(Confidentiality): 인가된 사용자만 접근 가능
    나. 무결성(Integrity): 불법적 변경으로부터 보호
    다. 가용성(Availability): 합법적 사용자의 서비스 접근 보장"

필요한 데이터 구조:
- 상위 개체(정보보안) → 노드
- 하위 구성요소 3개 → 각각 노드
- "구성요소" 관계 → HAS_COMPONENT 관계
```

**패턴 3: 비교 질문 → "두 개체 + 비교 관점"**
```
Q: "AES와 DES를 비교하시오."
A:
| 관점 | AES | DES |
|------|-----|-----|
| 키 길이 | 128/192/256bit | 56bit |
| 블록 크기 | 128bit | 64bit |
| 구조 | SPN | Feistel |
| 보안성 | 안전 | 취약 |

필요한 데이터 구조:
- 두 개체(AES, DES) → 각각 노드 (Technology)
- 비교 관계 → COMPARED_WITH 관계
- 비교 관점 목록 → 관계의 perspectives 속성
```

**패턴 4: 위협/대응 질문 → "위협 + 대상 + 대응"**
```
Q: "DDoS 공격의 유형과 대응방안을 설명하시오."
A: "DDoS는 서비스 가용성을 위협하며,
    방화벽/IPS/CDN 등으로 대응할 수 있다."

필요한 데이터 구조:
- 위협(DDoS) → Threat 노드
- 대상(가용성) → Concept 노드
- 대응(방화벽) → Technology 노드
- 관계: THREATENS, COUNTERED_BY
```

> 💡 **패턴 분석의 핵심**: 답변 문장에서 **명사**는 노드 후보, **동사/전치사**는 관계 후보입니다.
>
> "딥러닝**은** 머신러닝**의 하위 분야**이다" → (딥러닝)-[:SUBCLASS_OF]->(머신러닝)
> "DDoS**는** 가용성**을 위협**한다" → (DDoS)-[:THREATENS]->(가용성)
> "방화벽**으로** DDoS**를 대응**한다" → (DDoS)-[:COUNTERED_BY]->(방화벽)

---

#### ③ 노드 후보 식별

답변 패턴에서 반복적으로 등장하는 명사를 추출하고, 이를 노드 타입으로 분류합니다.

**추출 기준:**
- 여러 질문에서 반복 등장하는 명사 → 노드 후보
- 다른 명사와 관계를 맺는 명사 → 노드 후보
- 단순히 설명/수식하는 형용사적 표현 → 속성 후보

우리 프로젝트에서 추출한 노드 후보:

```
질문에서 반복 등장하는 명사:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
"딥러닝", "머신러닝", "교착상태" ...     → 개념 (Concept)
"Docker", "AES", "TCP" ...              → 기술 (Technology)
"애자일", "K-means", "블랙박스테스트" ... → 방법론 (Method)
"ISO 27001", "IEEE 802.11" ...          → 표준 (Standard)
"개인정보보호법", "GDPR" ...             → 법률 (Law)
"랜섬웨어", "DDoS", "SQL Injection" ... → 위협 (Threat)
"F1-Score", "가용성 99.99%" ...         → 지표 (Metric)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

이때 **정량 분석 데이터**가 노드 타입 분리의 근거가 됩니다:
- 개인정보보호 553회 출제 → Law 노드를 Concept에서 분리할 가치 충분
- 테스트 922회 최다 빈출 → Method.category에 test_method 추가
- 취약점 223회 + AI+SE 교차 34건 → Threat 독립 노드 타당

(노드 타입 설계의 세부 기법은 Ch.06에서 다룹니다.)

---

#### ④ 관계 패턴 도출

답변 패턴의 동사/전치사에서 관계를 도출합니다.

**문제 유형 → 관계 매핑:**

| 문제 유형 | 비중 | 핵심 동사/표현 | 도출된 관계 |
|----------|------|---------------|------------|
| 단답설명형 | 57.4% | "~의 한 종류", "~에 속하는" | SUBCLASS_OF, BELONGS_TO |
| 서술형(3항목) | 20.2% | "~로 구성", "~의 단계" | HAS_COMPONENT, HAS_PHASE |
| 비교형 | 11.3% | "A와 B를 비교" | COMPARED_WITH |
| 보안 관련 | 18.7% | "~를 위협", "~로 대응" | THREATENS, COUNTERED_BY |
| 기술 관련 | - | "~를 구현", "~에 의존" | IMPLEMENTS, DEPENDS_ON |
| 표준 관련 | - | "~에 의해 정의" | DEFINED_BY |
| 지표 관련 | - | "~로 측정" | MEASURED_BY |

**교차 출제 분석에서 추가 도출:**
- OS/CA + SW 교차 62건 → 도메인 간 공유 노드 필요 → BELONGS_TO가 다중 가능해야
- AI + SE 교차 34건 → Threat.cross_domains 속성 필요
- DB + SW 교차 42건 → 트랜잭션, 정규화 등 공유 개념 확인

(관계 설계의 세부 기법은 Ch.06에서 다룹니다.)

---

#### ⑤ 속성 확정

노드와 관계가 결정되면, 각각에 어떤 메타데이터(속성)가 필요한지 결정합니다.

**속성 결정 기준:**
```
이 속성이 질문의 답변에 직접 사용되는가?  → 필수 속성
이 속성으로 검색/필터링하는 경우가 있는가? → 인덱스 대상
이 속성이 나중에 필요할 수 있는가?         → 선택 속성
```

**질문 유형 → 속성 매핑:**

| 질문 유형 | 답변에 필요한 정보 | 도출된 속성 |
|----------|-------------------|------------|
| "X를 설명하시오" | X의 정의 | definition (필수) |
| "X를 설명하시오" | X의 한글명 | name_kr (필수) |
| "도메인별 출제 빈도" | 출제 횟수 | exam_frequency |
| "A와 B를 비교하시오" | 비교 관점 | perspectives (관계 속성) |
| "출제 트렌드 분석" | 시험 회차, 년도 | session, year |

(속성 설계의 세부 기법은 Ch.06에서 다룹니다.)

### 5.2.3 QDM 실전 적용 — 우리 프로젝트

자, 이제 QDM 5단계를 우리 프로젝트에 실제로 적용해봅시다.
10개의 대표 질문을 선정하고, 각 질문에서 스키마 요소를 도출하는 과정을 보여드리겠습니다.

---

**Q1: "딥러닝의 하위 기술에는 무엇이 있는가?"**

```
답변 패턴: 딥러닝 → (하위) → CNN, RNN, GAN, Transformer ...
도출:
  - 노드: Concept (딥러닝, CNN, RNN, GAN, Transformer)
  - 관계: SUBCLASS_OF (하위→상위 방향)

Cypher:
  MATCH (sub)-[:SUBCLASS_OF]->(c:Concept {id: 'deep_learning'})
  RETURN sub.name_kr
```

**Q2: "AES와 DES를 비교하시오."**

```
답변 패턴: AES ↔ DES, 비교 관점: [키 길이, 블록 크기, 구조, 보안성]
도출:
  - 노드: Technology (AES, DES)
  - 관계: COMPARED_WITH + perspectives 속성

Cypher:
  MATCH (a:Technology {id: 'aes'})-[r:COMPARED_WITH]-(b:Technology {id: 'des'})
  RETURN r.perspectives, r.summary
```

**Q3: "SDLC의 단계를 설명하시오."**

```
답변 패턴: SDLC → [요구분석, 설계, 구현, 테스트, 유지보수]
도출:
  - 노드: Concept (SDLC, 각 단계)
  - 관계: HAS_PHASE (순서 있는 구성)

Cypher:
  MATCH (sdlc:Concept {id: 'sdlc'})-[:HAS_PHASE]->(phase)
  RETURN phase.name_kr
```

**Q4: "DDoS 공격의 대응방안을 설명하시오."**

```
답변 패턴: DDoS → 위협 → 가용성, DDoS → 대응 → 방화벽, IPS, CDN
도출:
  - 노드: Threat (DDoS), Concept (가용성), Technology (방화벽, IPS, CDN)
  - 관계: THREATENS, COUNTERED_BY

Cypher:
  MATCH (t:Threat {id: 'ddos'})-[:COUNTERED_BY]->(counter)
  RETURN counter.name_kr
```

**Q5: "ISO 27001이 정의하는 보안 영역은?"**

```
답변 패턴: ISO 27001 → 정의 → [접근통제, 암호화, 물리보안 ...]
도출:
  - 노드: Standard (ISO 27001), Concept (보안 영역들)
  - 관계: DEFINED_BY (개념 → 표준)

Cypher:
  MATCH (c)-[:DEFINED_BY]->(s:Standard {id: 'iso_27001'})
  RETURN c.name_kr
```

**Q6: "머신러닝의 유형을 설명하시오."**

```
답변 패턴: 머신러닝 → [지도학습, 비지도학습, 강화학습]
도출:
  - 노드: Concept (머신러닝), Method (지도학습, 비지도학습, 강화학습)
  - 관계: HAS_TYPE

Cypher:
  MATCH (ml:Concept {id: 'machine_learning'})<-[:HAS_TYPE]-(type)
  RETURN type.name_kr
```

**Q7: "개인정보보호법의 주요 내용을 설명하시오."**

```
답변 패턴: 개인정보보호법 → [수집제한, 동의 원칙, 파기 의무 ...]
도출:
  - 노드: Law (개인정보보호법), Concept (각 원칙)
  - 관계: DEFINED_BY (원칙 → 법률)

Cypher:
  MATCH (c)-[:DEFINED_BY]->(law:Law {id: 'privacy_act'})
  RETURN c.name_kr
```

**Q8: "블록체인이 사용하는 핵심 기술은?"**

```
답변 패턴: 블록체인 → 사용 → [해시함수, 합의 알고리즘, P2P ...]
도출:
  - 노드: Concept (블록체인), Technology (해시함수, P2P), Method (합의 알고리즘)
  - 관계: USES, DEPENDS_ON

Cypher:
  MATCH (bc:Concept {id: 'blockchain'})-[:USES]->(tech)
  RETURN tech.name_kr
```

**Q9: "보안 도메인에서 가장 많이 출제된 개념 Top 10은?"**

```
답변 패턴: SE 도메인 → 소속 개념 → 출제 빈도 정렬
도출:
  - 노드: Domain (SE), Concept (보안 개념들)
  - 관계: BELONGS_TO
  - 속성: exam_frequency (정렬 기준)

Cypher:
  MATCH (c:Concept)-[:BELONGS_TO]->(d:Domain {id: 'SE'})
  RETURN c.name_kr, c.exam_frequency
  ORDER BY c.exam_frequency DESC LIMIT 10
```

**Q10: "SLA의 핵심 지표를 설명하시오."**

```
답변 패턴: SLA → 측정 → [가용성, 응답시간, 처리량 ...]
도출:
  - 노드: Concept (SLA), Metric (가용성, 응답시간, 처리량)
  - 관계: MEASURED_BY

Cypher:
  MATCH (sla:Concept {id: 'sla'})-[:MEASURED_BY]->(m:Metric)
  RETURN m.name_kr, m.unit
```

---

**10개 질문에서 도출된 스키마 요소 종합:**

| 도출 요소 | 수량 | 상세 |
|----------|------|------|
| 노드 타입 | 7개 | Concept, Technology, Method, Standard, Law, Threat, Metric |
| 관계 타입 | 11개 | SUBCLASS_OF, COMPARED_WITH, HAS_PHASE, THREATENS, COUNTERED_BY, DEFINED_BY, HAS_TYPE, USES, DEPENDS_ON, BELONGS_TO, MEASURED_BY |
| 노드 속성 | 5개 | id, name, name_kr, definition, exam_frequency |
| 관계 속성 | 2개 | perspectives, summary |

단 10개의 질문에서 **실제 스키마의 핵심 구조가 거의 완성**되었습니다.
나머지 4개 노드(Domain, Question, Document, ExamSession)는 추적/관리 목적의 구조 노드이며,
나머지 8개 관계도 추가 질문 분석에서 자연스럽게 도출됩니다.

> 💡 **QDM의 위력**: 데이터를 한 줄도 보지 않고, 질문 10개만으로 스키마의 80%를 설계했습니다.
> 이것이 "질문이 스키마를 결정한다"의 의미입니다.

### 5.2.4 [기업 박스] 비즈니스 질문에서 QDM 적용

> **기업 환경에서도 QDM은 동일하게 적용됩니다.**
> 차이점은 질문의 출처가 "기출문제"가 아니라 "비즈니스 요구사항"이라는 것뿐입니다.
>
> **시나리오: IT 인프라 지식그래프**
>
> | 비즈니스 질문 | 도출 요소 |
> |-------------|----------|
> | "이 서버가 다운되면 영향받는 서비스는?" | (:Service)-[:DEPENDS_ON]->(:Server) |
> | "이 취약점에 노출된 시스템 목록은?" | (:System)-[:HAS_VULNERABILITY]->(:CVE) |
> | "이 패치가 적용되지 않은 서버는?" | (:Server)-[:MISSING_PATCH]->(:Patch) |
> | "이 팀이 관리하는 인프라 전체 목록은?" | (:Team)-[:MANAGES]->(:Infrastructure) |
>
> **시나리오: 제품 추천 그래프**
>
> | 비즈니스 질문 | 도출 요소 |
> |-------------|----------|
> | "이 고객이 구매한 상품과 유사한 상품은?" | (:Customer)-[:PURCHASED]->(:Product)-[:SIMILAR_TO]->(:Product) |
> | "이 상품을 구매한 사람들이 함께 산 상품은?" | 경로 탐색: 공동 구매 패턴 |
> | "이 카테고리에서 가장 인기 있는 상품은?" | (:Product)-[:BELONGS_TO]->(:Category) + 정렬 |
>
> 핵심은 동일합니다:
> **"이해관계자에게 질문 리스트를 받아라 → 그 질문에 답하는 구조를 설계하라"**

## 5.3 모델링 접근법 비교

QDM은 "질문에서 출발"하는 방법론이지만,
실제 프로젝트에서는 여러 접근법을 조합하여 사용합니다.
대표적인 3가지 접근법을 비교하겠습니다.

### 5.3.1 Top-Down: 도메인 지식에서 출발

**방법**: 도메인 전문가의 지식과 이론에서 출발하여 구조를 정의한 후, 데이터를 맞춰 넣습니다.

```
전문가 지식 → 개념 체계 정의 → 관계 정의 → 데이터 적재
```

**장점:**
- 이론적으로 완결된 구조
- 도메인의 핵심 개념을 빠짐없이 포함
- 표준 온톨로지(SNOMED CT, FIBO 등)에 적합

**단점:**
- 현실 데이터와 괴리 가능 ("이론상 존재하지만 데이터에는 없는" 노드)
- 전문가 의존도가 높음
- 과도하게 복잡한 구조를 만들기 쉬움

**적합한 경우**: 해당 도메인에 이미 확립된 체계가 있을 때 (의학, 법률 등)

### 5.3.2 Bottom-Up: 데이터에서 출발

**방법**: 원본 데이터를 분석하여 패턴을 발견하고, 발견된 패턴으로 구조를 만듭니다.

```
데이터 수집 → 패턴 분석 → 구조 도출 → 정제/확장
```

**장점:**
- 현실 데이터에 밀착된 구조
- 실증적 근거 (빈도, 분포 기반)
- 누락 위험 적음 (데이터에 있는 것을 구조화)

**단점:**
- 구조 없는 스파게티 그래프 위험 ("데이터를 그냥 다 넣은" 상태)
- 데이터 노이즈가 구조에 반영될 수 있음
- 도메인 지식 없이는 의미 없는 패턴을 구조화할 위험

**적합한 경우**: 대량의 비정형 데이터를 구조화해야 할 때

### 5.3.3 Middle-Out: QDM의 접근

**방법**: 질문(사용 맥락)에서 출발하여, 위로(도메인 이론)와 아래로(데이터) 양방향으로 확장합니다.

```
핵심 질문 → 스키마 도출 → 위로: 이론 보완 + 아래로: 데이터 검증
```

**장점:**
- 실용적이면서 구조적
- 사용 목적에 최적화된 구조
- 이론과 데이터의 균형

**단점:**
- 초기 질문이 빠지면 해당 영역이 누락될 수 있음
- 질문이 변하면 구조도 변해야 함 (하지만 이것은 장점이기도 함)

**적합한 경우**: 대부분의 실무 프로젝트 (특히 지식그래프)

### 5.3.4 우리 프로젝트의 접근: Bottom-Up + Middle-Out 하이브리드

우리 프로젝트는 두 접근법을 결합했습니다:

```
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  Phase 1: Bottom-Up (정량 분석)                               │
│  ─────────────────────────────                                │
│  22,518줄 기출문제 전수 분석                                   │
│  → 150개 키워드 빈도 카운트                                    │
│  → 도메인 교차 출제 패턴 발견                                  │
│  → 문제 유형 분포 확인                                        │
│  → "무엇이 중요한가" 판단                                     │
│                                                              │
│  Phase 2: Middle-Out (QDM)                                   │
│  ────────────────────────                                     │
│  문제 유형 = 질문 유형으로 치환                                │
│  → 각 질문 유형에서 답변 패턴 분석                             │
│  → 노드/관계/속성 도출                                        │
│  → "어떻게 구조화하나" 결정                                    │
│                                                              │
│  결합 효과:                                                   │
│  정량 분석이 "중요도"를 제공하고                               │
│  QDM이 "구조"를 제공합니다                                    │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**구체적으로:**

| 단계 | 접근법 | 수행 내용 | 산출물 |
|------|--------|----------|--------|
| 1 | Bottom-Up | 기출문제 22,518줄 정량 분석 | 키워드 Top 50, 도메인 비중, 교차 패턴 |
| 2 | Middle-Out (QDM) | 문제 유형에서 질문 도출 | 5가지 질문 유형, 10개 대표 질문 |
| 3 | Middle-Out (QDM) | 질문에서 스키마 도출 | 노드 11개, 관계 19개, 속성 규칙 |
| 4 | Bottom-Up | 정량 분석으로 우선순위 결정 | Top 50 시드 노드, exam_weight |
| 5 | 검증 | 표본 적재 후 쿼리 테스트 | 스키마 v2 (정량 반영) |

이 하이브리드 접근법의 핵심 이점:
- 정량 분석이 "감(感)" 대신 **데이터 기반** 의사결정을 가능하게 합니다
- QDM이 데이터를 단순히 "넣는" 대신 **목적에 맞게 구조화**합니다
- 둘의 결합으로 "중요한 것을 올바른 구조로" 표현할 수 있습니다

## 5.4 스키마 설계의 보편 원칙

이 절에서는 QDM과 모델링 접근법을 초월하여,
**모든 그래프 스키마 설계에 적용되는 보편 원칙**을 정리합니다.

### 5.4.1 [설계 원칙] 첫 스키마는 틀린다

이것은 겸손의 표현이 아닙니다. **구조적 진실**입니다.

```
왜 첫 스키마가 틀리는가:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. 데이터를 전부 보기 전에 설계하므로, 예외 케이스를 모름
2. 실제 쿼리를 돌려보기 전에 설계하므로, 성능 문제를 모름
3. 도메인 지식이 설계 과정에서 점진적으로 깊어지므로, 초기 이해가 불완전
4. 사용자 요구사항이 시간이 지나면 변하므로, 고정된 구조는 없음
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**그래서 어떻게 해야 하는가?**

완벽한 설계를 추구하면 영원히 시작할 수 없습니다. 대신:

1. **80% 설계로 시작**: 핵심 질문의 80%에 답할 수 있는 수준이면 충분합니다
2. **빠르게 적재**: 표본 데이터 10~50개를 즉시 적재합니다
3. **쿼리로 검증**: 설계한 질문들을 실제로 Cypher로 실행합니다
4. **발견 → 수정**: 문제를 발견하면 즉시 수정합니다
5. **반복**: 이 사이클을 3~4회 반복하면 스키마가 안정화됩니다

Neo4j의 **스키마리스(schema-less)** 특성이 이 전략을 가능하게 합니다:
- 새 라벨 추가: 그냥 만들면 됩니다
- 새 관계 추가: 기존 노드 사이에 자유롭게 연결
- 속성 추가: SET으로 즉시 추가
- RDB처럼 ALTER TABLE, 마이그레이션 스크립트가 필요 없습니다

> 💡 **실화**: 우리 프로젝트에서도 첫 스키마(v1)는 부족했습니다.
> 정량 분석을 반영하여 v2로 수정했고, 검증(STEP 5) 후 v3으로 진화했습니다.
> v1 → v2에서 변경된 것: exam_weight 속성 추가, test_method 카테고리 추가, cross_domains 추가
> v2 → v3에서 변경된 것: 중복 노드 병합, 교차 라벨 이슈 해결, MENTIONED_IN 보강

### 5.4.2 [설계 원칙] MERGE 기반 성장

Neo4j에서 온톨로지를 구축할 때 가장 중요한 단일 키워드가 있다면, **MERGE**입니다.

```cypher
// CREATE는 매번 새 노드를 만듭니다 (중복 위험!)
CREATE (c:Concept {id: 'deep_learning', name: 'Deep Learning'})
// → 실행할 때마다 새 노드 생성 → 10번 실행하면 10개 중복!

// MERGE는 "있으면 재사용, 없으면 생성"합니다 (중복 방지!)
MERGE (c:Concept {id: 'deep_learning'})
SET c.name = 'Deep Learning', c.name_kr = '딥러닝'
// → 100번 실행해도 노드는 1개만 존재
```

MERGE가 온톨로지에서 특히 중요한 이유:

**1. 중복 방지**
- 같은 개념이 여러 문서에서 등장합니다
- AI_001에서 "딥러닝"을 만들고, AI_003에서 또 "딥러닝"을 만들면?
- CREATE: 2개 노드 → 관계가 분산되어 의미 손실
- MERGE: 1개 노드 → 모든 관계가 하나로 수렴

**2. 도메인 간 자동 연결**
```cypher
// AI 도메인 적재 시
MERGE (c:Concept {id: 'encryption'})  // 이미 SE 도메인에서 만들어진 노드
SET c.name = 'Encryption', c.name_kr = '암호화'
// → SE 도메인의 관계와 AI 도메인의 관계가 하나의 노드에 수렴
// → 도메인 간 연결이 "자동으로" 생김!
```

**3. 시드 노드 전략과 결합**
```cypher
// 00_schema.cypher에서 시드 생성
MERGE (c:Concept {id: 'encryption'})
SET c.exam_frequency = 212, c.importance = 'high'

// 나중에 se_001_010.cypher에서 문서 데이터 적재
MERGE (c:Concept {id: 'encryption'})
SET c.definition = '정보를 해독할 수 없도록 변환하는 기술'
// → 시드의 exam_frequency + 문서의 definition이 하나의 노드에 합쳐짐
```

> ⚠️ **MERGE의 함정**: MERGE는 **id로만 매칭**해야 합니다.
> ```cypher
> // 잘못된 MERGE — 모든 속성으로 매칭
> MERGE (c:Concept {id: 'deep_learning', name: 'Deep Learning', domain: 'AI'})
> // → name이나 domain이 조금이라도 다르면 새 노드 생성!
>
> // 올바른 MERGE — id로만 매칭, 나머지는 SET
> MERGE (c:Concept {id: 'deep_learning'})
> SET c.name = 'Deep Learning', c.domain = 'AI'
> ```

### 5.4.3 [설계 원칙] 질문이 바뀌면 스키마도 바뀐다

스키마는 완성되는 것이 아니라 **진화**하는 것입니다.

```
Phase 1 (현재):
  질문: "이 개념의 정의는?" "하위 기술은?" "비교대상은?"
  → 스키마: Concept, Technology, Method + 기본 관계 17개

Phase 2 (예정):
  새 질문: "이 문제의 정답 코드는?" "이 공식을 유도하시오."
  → 스키마 확장: CodePattern 노드, Formula 노드 추가

Phase 3 (미래):
  새 질문: "이 개념이 시대별로 어떻게 변했는가?"
  → 스키마 확장: ExamSession 활용, 시간 기반 관계 추가
```

이것은 결함이 아니라 **건강한 진화**입니다.
처음부터 Phase 3까지 설계하려고 하면:
- 복잡성이 폭발합니다
- 실제로 사용하지 않는 구조가 만들어집니다
- YAGNI(You Ain't Gonna Need It) 원칙 위반입니다

> 📌 **[설계 원칙] 현재 질문에만 답하는 스키마를 만들어라.**
> **미래 질문에 대한 준비는 "확장 가능한 구조"로 충분하다.**
>
> Neo4j의 스키마리스 특성 덕분에, 나중에 노드/관계를 추가하는 것은
> RDB에서 테이블을 추가하는 것보다 훨씬 쉽습니다.

## 5.5 흔한 함정과 실수

스키마 설계 과정에서 초보자가 빠지기 쉬운 함정을 정리합니다.

### 함정 1: "데이터부터 넣고 구조는 나중에"

```
❌ 잘못된 접근:
   "일단 2,392개 문서를 다 파싱해서 넣자.
    구조는 나중에 정리하면 되잖아."

✅ 올바른 접근:
   "10개 문서를 먼저 분석해서 스키마를 설계하고,
    검증한 후에 규모를 키우자."
```

왜 위험한가:
- 구조 없이 데이터를 넣으면 "거대한 스파게티 그래프"가 됩니다
- 나중에 구조를 부여하려면, 이미 적재된 데이터를 전부 재처리해야 합니다
- 10개에서 발견할 수 있는 문제를 1,000개 적재 후 발견하면 수정 비용이 100배입니다

### 함정 2: "완벽한 설계를 먼저 만들자"

```
❌ 잘못된 접근:
   "온톨로지 이론을 다 공부하고, OWL로 형식화하고,
    모든 예외 케이스를 다 고려한 후에 시작하자."

✅ 올바른 접근:
   "핵심 질문 10개에 답할 수 있는 수준으로 설계하고,
    표본 적재 후 검증하면서 개선하자."
```

왜 위험한가:
- 완벽을 추구하면 영원히 시작할 수 없습니다
- 데이터를 실제로 적재해보기 전에는 진짜 문제를 알 수 없습니다
- 시간의 80%를 설계에 쓰고 20%를 구현에 쓰면, 설계 품질이 떨어집니다
  (검증 피드백 없이 머릿속에서만 설계하므로)

### 함정 3: "모든 것을 하나의 노드 타입으로"

```
❌ God Node 안티패턴:
   (:Entity {type: 'concept', name: '딥러닝'})
   (:Entity {type: 'technology', name: 'Docker'})
   (:Entity {type: 'threat', name: 'DDoS'})
   → 모든 것이 Entity, type 속성으로만 구분

✅ 올바른 접근:
   (:Concept {name: '딥러닝'})
   (:Technology {name: 'Docker'})
   (:Threat {name: 'DDoS'})
   → 라벨로 명확히 구분
```

왜 위험한가:
- 제약조건을 타입별로 걸 수 없습니다
- 쿼리 시 항상 WHERE type = '...' 필터가 필요합니다
- 인덱스 효율이 떨어집니다
- Neo4j의 라벨 기반 최적화를 활용할 수 없습니다

### 함정 4: "관계를 너무 세분화"

```
❌ 관계 폭발:
   USED_IN_AI_DOMAIN
   USED_IN_SE_DOMAIN
   USED_IN_NW_DOMAIN
   → 도메인마다 별도 관계 타입

✅ 올바른 접근:
   BELONGS_TO + Domain 노드
   → 하나의 관계 타입 + 대상 노드로 구분
```

왜 위험한가:
- 도메인이 추가될 때마다 새 관계 타입을 정의해야 합니다
- 쿼리 시 관계 타입을 모두 나열해야 합니다: `[:USED_IN_AI|USED_IN_SE|...]`
- 관계의 의미가 같은데 이름만 다른 것은 혼란을 가중합니다

## 5.5 OntoClean으로 노드 분류 검증하기

QDM으로 노드 후보를 도출했다면, 그 분류가 **철학적으로 올바른지** 검증할 방법이 필요합니다.
**OntoClean**(Guarino & Welty, 2002, 2004)은 온톨로지의 택소노미 구조를 검증하는
가장 체계적인 방법론으로, 형식 존재론(formal ontology)의 메타속성을 활용합니다.

> 💡 OntoClean은 "분류가 맞는가?"를 판단하는 도구입니다.
> QDM이 "무엇을 만들까"를 결정한다면, OntoClean은 "만든 것이 논리적으로 건전한가"를 검증합니다.

### 5.5.1 4가지 메타속성 (Meta-Properties)

OntoClean은 각 클래스(= 노드 타입)에 대해 4가지 철학적 질문을 던집니다:

| 메타속성 | 질문 | 기호 | 예시 |
|----------|------|------|------|
| **강성 (Rigidity)** | "이 속성을 잃으면 그 개체가 아닌가?" | +R / −R / ~R | 사람은 항상 사람(+R), 학생은 졸업하면 아님(−R) |
| **정체성 (Identity)** | "이 클래스가 개체를 구별하는 기준을 제공하는가?" | +I / −I | 사람(주민번호, +I), 빨간 것(기준 없음, −I) |
| **통일성 (Unity)** | "이 클래스의 개체는 통일된 전체인가?" | +U / −U | 유기체(+U), 지리 영역(경계가 모호, −U) |
| **의존성 (Dependence)** | "이 클래스의 존재가 다른 개체에 의존하는가?" | +D / −D | 학생은 학교에 의존(+D), 사람은 독립(−D) |

### 5.5.2 우리 프로젝트에 적용: 11개 노드 타입 검증

| 노드 타입 | R | I | U | D | 검증 결과 |
|-----------|---|---|---|---|-----------|
| **Concept** | +R | +I | +U | −D | **독립 엔티티** — 딥러닝은 항상 딥러닝. id로 식별. 다른 개체와 무관하게 존재 |
| **Technology** | −R | +I | +U | +D | **구현체** — CNN은 딥러닝(Concept)에 의존. 이론이 바뀌면 기술도 변할 수 있음(−R) |
| **Method** | −R | +I | +U | +D | **절차/알고리즘** — 특정 목적에 의존(+D). SVM은 분류 목적이 사라지면 의미 약화(−R) |
| **Standard** | +R | +I | +U | +D | **규격** — ISO 27001은 항상 ISO 27001(+R). 발행 기관에 의존(+D) |
| **Law** | +R | +I | +U | +D | **법률** — 개인정보보호법은 항상 그 법(+R). 국가/제도에 의존(+D) |
| **Threat** | −R | +I | +U | +D | **위협** — DDoS는 대응 기술에 의해 무력화 가능(−R). 대상에 의존(+D) |
| **Metric** | +R | +I | −U | +D | **지표** — F1-Score는 항상 F1-Score(+R). 경계가 맥락 의존적(−U). 측정 대상에 의존(+D) |
| **Domain** | +R | +I | +U | −D | **분류 카테고리** — AI는 항상 AI(+R). 독립적으로 존재(−D) |
| **Document** | +R | +I | +U | −D | **원본 추적** — AI_001은 항상 그 문서(+R). 물리 파일 자체로 존재(−D) |
| **Question** | +R | +I | +U | +D | **기출문제** — 특정 시험에 의존(+D) |
| **ExamSession** | +R | +I | +U | −D | **시험 회차** — 132회는 항상 132회(+R). 독립 존재(−D) |

### 5.5.3 OntoClean 제약조건으로 스키마 검증

OntoClean의 핵심 제약조건:

> **"강성이 아닌 클래스(−R)는 강성 클래스(+R)의 상위 클래스가 될 수 없다."**

이것을 우리 프로젝트의 SUBCLASS_OF 관계에 적용하면:

```
✅ 올바름: (Technology:−R)-[:SUBCLASS_OF]->(Concept:+R)
   CNN은 −R, 딥러닝은 +R → "구현체가 개념의 하위"는 논리적으로 타당

❌ 위반 사례 (만약 이렇게 설계했다면):
   (Concept:+R)-[:SUBCLASS_OF]->(Method:−R)
   "딥러닝이 SVM의 하위"? → 개념(항상 존재)이 방법(맥락 의존)의 하위? 위반!
```

> 📌 **[설계 원칙] OntoClean 리트머스 테스트**
>
> SUBCLASS_OF 관계를 만들 때 이 질문을 던지세요:
> "A가 B의 하위라면, A는 B보다 **덜 근본적(less rigid)**인가?"
> - 답이 "예" → 올바른 계층
> - 답이 "아니오" → SUBCLASS_OF 대신 IMPLEMENTS, USES, HAS_TYPE 등 검토

### 5.5.4 Concept vs Technology vs Method — 3자 구분의 이론적 근거

QDM과 OntoClean을 결합하면, 이 3자 구분이 **직관이 아닌 논리**에 기반함을 증명할 수 있습니다:

```
┌────────────────────────────────────────────────────────────────┐
│                                                                │
│  Concept (+R, −D): "그 자체로 존재하는 개념"                   │
│  → 딥러닝, 교착상태, 정규화 — 구현과 무관하게 존재             │
│                                                                │
│  Technology (−R, +D): "개념을 구체적으로 실현한 것"            │
│  → CNN, AES, Docker — 개념에 의존하여 존재                     │
│                                                                │
│  Method (−R, +D): "목적을 달성하기 위한 절차"                  │
│  → SVM, K-Means, 애자일 — 목적/맥락에 의존하여 존재            │
│                                                                │
│  ※ Technology와 Method의 차이:                                 │
│  Technology = "무엇으로?" (도구/산출물)                         │
│  Method = "어떻게?" (절차/과정)                                │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

> ⚠️ **[안티패턴] "Technology와 Method를 합치면 안 되나?"**
>
> OntoClean 관점에서 Technology와 Method는 모두 −R, +D이므로 메타속성만으로는
> 구분이 불명확합니다. 그러나 **QDM 관점에서** 구분이 필수적입니다:
> - "이미지 분류에 사용되는 **기술**은?" → Technology (CNN, ResNet)
> - "클러스터링 **방법론**을 비교하시오" → Method (K-Means, DBSCAN)
>
> 같은 질문에서 도구(What)와 절차(How)가 혼재하면
> 답변이 부정확해집니다. **질문 유형이 다르면 노드 타입도 다릅니다.**

## 5.6 Ontology Design Patterns (ODPs) — 검증된 모델링 해법

### 5.6.1 ODP란 무엇인가

Ontology Design Patterns(ODPs)는 소프트웨어 공학의 디자인 패턴과 같은 개념입니다.
**반복적으로 등장하는 모델링 문제에 대한 검증된 해법**으로,
Gangemi(2005)와 ontologydesignpatterns.org 커뮤니티에 의해 체계화되었습니다.

우리 프로젝트에서 이미 사용하고 있는 패턴들을 ODP 관점에서 정리하면:

### 5.6.2 Content ODPs — 우리 프로젝트의 패턴 매핑

| ODP 이름 | 설명 | 우리 프로젝트의 구현 | 관계 |
|----------|------|---------------------|------|
| **Taxonomy** | 상하위 계층 분류 | Concept→Concept 체인 | SUBCLASS_OF |
| **Part-Whole** | 전체-부분 관계 | 정보보호→기밀성, 무결성, 가용성 | HAS_COMPONENT |
| **Classification** | 인스턴스-클래스 관계 | CNN→딥러닝 (구현체→개념) | INSTANCE_OF |
| **Participation** | 개체의 역할 참여 | SVM→지도학습 (방법이 패러다임에 참여) | IMPLEMENTS |
| **Sequence** | 순서가 있는 단계 | SDLC→요구분석→설계→구현→테스트 | HAS_PHASE |
| **N-ary Relation** | 3개 이상 개체의 관계 | AES↔DES 비교 (perspectives 속성으로 관점 포함) | COMPARED_WITH |
| **Description** | 개념의 정의/설명 | Concept.definition 속성 | (속성) |
| **Collection** | 그룹핑/분류 | Domain 노드에 BELONGS_TO | BELONGS_TO |

### 5.6.3 N-ary Relation 패턴 심화 — COMPARED_WITH의 설계 근거

비교(Comparison)는 본질적으로 **3항 관계(N-ary Relation)**입니다:
"A와 B를, C의 관점에서 비교한다." 이것을 Property Graph에서 표현하는 3가지 방법:

```
방법 1: 관계 속성 (우리 프로젝트의 선택)
(AES)-[:COMPARED_WITH {perspectives: ['키 길이', '속도', '안전성']}]->(DES)
장점: 간결, 쿼리 쉬움
단점: 관점별 상세 비교 불가

방법 2: 중간 노드 (Reification)
(AES)-[:COMPARED_IN]->(c:Comparison {topic: '대칭키'})<-[:COMPARED_IN]-(DES)
(c)-[:HAS_PERSPECTIVE]->(:Perspective {name: '키 길이', aes: '128/256bit', des: '56bit'})
장점: 관점별 상세 데이터 저장 가능
단점: 노드 폭발, 쿼리 복잡

방법 3: 하이브리드 (v2 이후 확장 가능)
현재는 방법 1, 필요 시 방법 2로 진화
```

> 📌 **[설계 원칙] "단순한 것에서 시작하고, 질문이 요구할 때 복잡하게 진화하라"**
>
> 현재 질문("AES와 DES를 비교하시오")은 관점 목록만 필요하므로 방법 1이면 충분합니다.
> "AES의 키 길이는 DES와 비교해 몇 배인가?"라는 질문이 나오면 방법 2로 진화합니다.
> 이것이 "첫 스키마는 틀린다"를 실천하는 방법입니다.

### 5.6.4 ODP 선택 가이드

새로운 관계를 설계할 때 어떤 ODP를 적용할지 결정하는 플로우차트:

```
┌─────────────────────────────────────────────────┐
│ "이 관계가 어떤 구조를 표현하는가?"              │
│                                                   │
│ 상하위?        → Taxonomy (SUBCLASS_OF)           │
│ 전체-부분?     → Part-Whole (HAS_COMPONENT)       │
│ 구현/실체?     → Classification (INSTANCE_OF)     │
│ 순서?          → Sequence (HAS_PHASE)             │
│ 비교?          → N-ary (COMPARED_WITH + 속성)     │
│ 분류/그룹?     → Collection (BELONGS_TO)          │
│ 대응/방어?     → Domain-specific (COUNTERED_BY)   │
│                                                   │
│ 위 중 해당 없음? → RELATED_TO (임시) + TODO 태그  │
│                   → 다음 사이클에서 구체화          │
└─────────────────────────────────────────────────┘
```

> ⚠️ **[안티패턴] Generic RELATED_TO 남용**
>
> ODP를 모르면 모든 관계를 RELATED_TO로 만드는 유혹에 빠집니다.
> RELATED_TO는 "아직 의미를 파악하지 못한 관계"의 임시 표식이어야 하며,
> 최종 스키마에 남아 있으면 안 됩니다. 이것을 **"관계 타입 빈곤(Relationship Type Poverty)"**이라 합니다.
