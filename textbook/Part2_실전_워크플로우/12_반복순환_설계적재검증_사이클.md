# Ch.12 — 반복순환: 설계→적재→검증 사이클

> **"첫 스키마는 틀린다. 두 번째도 틀린다. 세 번째쯤 쓸 만해진다."**

이 챕터에서는 STEP 3(설계) → STEP 4(적재) → STEP 5(검증)를
**하나의 순환 사이클**로 묶어, 스키마를 점진적으로 진화시키는 방법을 다룹니다.
선형적 프로세스가 아닌 **반복 개선**이 온톨로지 구축의 본질입니다.

---

## 12.1 왜 반복이 필요한가

### 12.1.1 "한 번에 완벽하게"는 불가능하다

온톨로지 설계가 RDB 스키마 설계와 근본적으로 다른 점:

| 특성 | RDB 스키마 | 온톨로지 |
|------|-----------|---------|
| **설계 시점** | 개발 초기에 확정 | 데이터 투입하면서 진화 |
| **변경 비용** | 높음 (마이그레이션) | 낮음 (스키마리스) |
| **정답 기준** | ERD 정규화 이론 | 질문에 답할 수 있는가? |
| **검증 방법** | 정규화 수준 확인 | 데이터 적재 후 검증 쿼리 |
| **완성 시점** | 테이블 생성 시 | 데이터가 충분히 쌓인 후 |

RDB는 "설계 → 구현 → 운영"의 선형 흐름입니다.
온톨로지는 "설계 → 적재 → 검증 → **재설계**"의 순환 흐름입니다.

### 12.1.2 우리 프로젝트의 실증 — v1에서 v2로

우리는 이미 한 번의 순환을 경험했습니다:

```
v1 스키마 (초기 설계)
├── 정량 분석 없이 직관으로 설계
├── 노드 타입 7개, 관계 타입 12개
├── 5개 도메인 46개 문서 적재
└── STEP 5 검증 결과: 5건 이슈 발견
    ├── Domain.name 영문 덮어쓰기 (3건)
    ├── 교차 라벨 중복 — function_point (1건)
    ├── id 네이밍 불일치 — agile (1건)
    ├── Top 50 시드 미연결 (45/50)
    └── BELONGS_TO 누락 (3건)
        ↓
v2 스키마 (정량 분석 반영)
├── 기출문제 22,518줄 전수 분석
├── 노드 타입 11개, 관계 타입 17개
├── Top 50 시드 노드 선정 + Domain 연결
├── 이슈 5건 전체 해결
└── 다음 사이클 준비 (100개 적재)
```

**v1→v2에서 바뀐 것:**

| 변경 항목 | v1 | v2 | 변경 근거 |
|----------|----|----|----------|
| 노드 타입 | 7개 | 11개 | Threat, Metric, Question, ExamSession 추가 |
| 관계 타입 | 12개 | 17개 | THREATENS, COUNTERED_BY 등 보안 관계 추가 |
| 시드 노드 | 없음 | Top 50 | 정량 분석 기반 |
| Domain 속성 | name만 | name + exam_weight + doc_count | 정량 분석 반영 |
| id 규칙 | 비일관적 | 소문자_언더스코어 강제 | 중복 방지 |

이 변화는 **데이터를 적재하고 검증한 후에야** 발견할 수 있었습니다.
머릿속 설계만으로는 절대 나올 수 없는 개선입니다.

## 12.2 반복 사이클 모델

### 12.2.1 사이클 구조

```
┌→ 설계(수정) → 적재(추가) → 검증 → 이슈 발견 ─┐
│                                              │
│  스키마 확정?  ← No ─────────────────────────┘
│       │
│      Yes
│       │
│       ↓
│  다음 사이클 (더 많은 데이터로)
│       │
└───────┘
```

**각 단계의 역할:**

| 단계 | 입력 | 작업 | 출력 | 기준 |
|------|------|------|------|------|
| **설계(수정)** | 이전 검증 이슈 | 스키마 변경, 규칙 강화 | 수정된 스키마 | 이슈 0건 |
| **적재(추가)** | JSON 문서 N개 | Cypher 생성 + 실행 | 그래프 데이터 | 오류 0건 |
| **검증** | 적재된 그래프 | L1~L5 진단 쿼리 | 이슈 목록 | 이슈 유형 분류 |
| **판단** | 이슈 목록 | 반복 필요 여부 결정 | Go/No-Go | 임계값 기반 |

### 12.2.2 사이클 종료 조건

무한 반복을 방지하기 위한 종료 조건:

```
사이클 종료 조건 (모두 충족 시):
├── ✅ L1(구조): 제약조건/인덱스 전수 통과
├── ✅ L2(데이터): 필수 속성 NULL 비율 < 5%
├── ✅ L3(관계): 고립 노드 0개, BELONGS_TO 누락 0건
├── ✅ L4(의미): 교차 라벨 중복 0건, 중복 id 0건
├── ✅ L5(성능): 주요 쿼리 응답 < 100ms
└── ✅ 신규 이슈 타입 없음 (기존 이슈 재발만 있을 경우)
```

> 💡 **"이슈 0건"이 아니라 "신규 이슈 타입 없음"이 기준인 이유**
>
> 데이터를 추가할 때마다 사소한 속성 누락은 발생할 수 있습니다.
> 중요한 것은 **새로운 종류의 구조적 문제**가 발생하지 않는 것입니다.
> 반복적인 사소한 이슈는 자동화(Cypher 작성 규칙)로 예방합니다.

## 12.3 사이클별 체크리스트

### Cycle 1: 표본 10개 — 기본 구조 검증

**목적**: 스키마의 기본 구조가 데이터를 담을 수 있는지 확인

```
적재량: 1~2개 도메인, 10개 문서
검증 초점:
├── 노드 타입이 데이터를 분류하기에 충분한가?
├── 관계 타입이 텍스트의 의미를 표현하는가?
├── id 네이밍 규칙이 일관적인가?
├── MERGE가 중복 없이 동작하는가?
└── Domain 연결이 올바른가?

예상 이슈: 노드 타입 부족, 관계 타입 부족, id 규칙 미확립
수정 범위: 스키마 구조 대폭 변경 가능 (데이터 적으므로)
```

**실전 체크리스트:**

- [ ] 모든 JSON 텍스트가 최소 1개 노드로 변환되었는가?
- [ ] 문서 간 공유 개념이 MERGE로 연결되었는가?
- [ ] 고립 노드가 없는가? (`MATCH (n) WHERE NOT (n)--() RETURN n`)
- [ ] Domain 노드의 name이 한글로 유지되는가?

### Cycle 2: 50개 — 패턴 안정화

**목적**: 스키마 패턴이 다양한 데이터에서도 안정적인지 확인

```
적재량: 3~5개 도메인, 50개 문서
검증 초점:
├── 교차 도메인 개념이 공유 노드로 연결되는가?
├── 시드 노드와 문서 데이터가 MERGE로 자연 연결되는가?
├── 관계 방향이 일관적인가?
├── 중복 관계가 발생하지 않는가?
└── 교차 라벨 문제가 없는가?

예상 이슈: id 충돌, 교차 라벨 중복, 시드-문서 연결 누락
수정 범위: 관계 추가/속성 변경 수준 (구조 변경은 최소화)
```

**실전 체크리스트:**

- [ ] Top 50 시드 중 적재 도메인과 관련된 시드가 문서와 연결되었는가?
- [ ] 같은 개념이 다른 id로 생성되지 않았는가? (agile vs agile_methodology)
- [ ] 비교형 관계(COMPARED_WITH)에 perspectives 속성이 있는가?
- [ ] 교차 라벨 검증: 동일 id로 Concept + Method가 동시 존재하지 않는가?

### Cycle 3: 100개 — 스키마 확정

**목적**: 스키마를 "확정"하고, 이후에는 구조 변경 없이 적재만 반복

```
적재량: 전 도메인, 100개 문서
검증 초점:
├── 13개 도메인 모두에서 스키마가 동작하는가?
├── 노드 수/관계 수 비율이 합리적인가? (1:2~1:3 기대)
├── 성능에 문제가 없는가? (PROFILE 확인)
├── Cypher 작성 규칙이 충분히 강화되었는가?
└── 자동화 가능한 부분이 식별되었는가?

예상 이슈: 성능 튜닝, 마이너 속성 추가
수정 범위: 속성 추가/인덱스 추가 수준 (구조 동결)
```

**스키마 확정의 의미:**

```
확정 = 다음을 변경하지 않겠다는 결정:
├── 노드 타입 (11개 고정)
├── 관계 타입 (17개 고정, 추가는 가능하되 삭제 불가)
├── id 네이밍 규칙 (영문 소문자 + 언더스코어)
├── 필수 속성 (id, name, name_kr, domain)
└── Cypher 작성 규칙 (MERGE only, Domain.name SET 금지)

확정해도 변경 가능한 것:
├── 추가 속성 (description, url, created_at 등)
├── 추가 인덱스 (성능 필요 시)
├── 추가 관계 타입 (새로운 패턴 발견 시)
└── 속성값 (definition, exam_frequency 등)
```

### Cycle 4: 전체 — 자동화 파이프라인

**목적**: 2,392개 문서 전체를 자동/반자동으로 적재

```
적재량: 전체 2,392개 문서
검증 초점:
├── 배치 적재 오류율 < 1%
├── 전체 그래프 통계가 기대치와 일치하는가?
├── 성능 테스트 (동시 쿼리, 대량 MATCH)
└── GraphRAG 확장 준비 (벡터 임베딩, 유사도 검색)

예상 이슈: 배치 처리 오류, 대규모 중복 검출, 성능 병목
수정 범위: 자동화 스크립트 개선 (스키마 변경 없음)
```

## 12.5 스키마 버전 관리 실전

### 12.5.1 버전 네이밍 규칙

```
v1.0: 초기 설계 (직관 기반)
v1.1: 표본 적재 후 미세 조정
v2.0: 정량 분석 반영 (대폭 변경)
v2.1: 50개 적재 후 조정
v3.0: 스키마 확정 버전
```

**Major 변경** (v1→v2): 노드 타입 추가/삭제, 관계 타입 구조 변경
**Minor 변경** (v2.0→v2.1): 속성 추가, 인덱스 추가, 규칙 보강

### 12.5.2 Cypher 마이그레이션 스크립트 패턴

스키마를 변경할 때는 **멱등성 있는 마이그레이션 스크립트**를 작성합니다:

```cypher
// ================================================================
// Migration: v2.0 → v2.1
// 날짜: 2026-03-15
// 변경 사항:
//   1. Concept에 category 속성 추가
//   2. RELATED_TO → DEPENDS_ON 구체화 (3건)
//   3. 인덱스 추가: Concept.category
// ================================================================

// --- 1. 속성 추가 (멱등: 이미 있으면 덮어씀) ---
MATCH (c:Concept)
WHERE c.category IS NULL
SET c.category = 'general';

// --- 2. 관계 타입 변경 (멱등: MERGE 사용) ---
MATCH (a:Concept {id: 'deep_learning'})-[r:RELATED_TO]->(b:Concept {id: 'gpu'})
MERGE (a)-[:DEPENDS_ON]->(b)
DELETE r;

// --- 3. 인덱스 추가 (멱등: IF NOT EXISTS) ---
CREATE INDEX concept_category IF NOT EXISTS
FOR (n:Concept) ON (n.category);

// --- 검증 ---
// MATCH (c:Concept) WHERE c.category IS NULL RETURN count(c);
// → 0이어야 함
```

> [설계 원칙] **"마이그레이션 스크립트는 몇 번을 실행해도 같은 결과"**
>
> - 속성 추가: `WHERE ... IS NULL` 조건 + SET
> - 인덱스: `IF NOT EXISTS`
> - 노드/관계: `MERGE` (이미 있으면 무시)
> - 삭제: `WHERE EXISTS` 조건 확인 후 DELETE
>
> 멱등성이 보장되면 스크립트를 안심하고 반복 실행할 수 있습니다.

### 12.5.3 롤백 전략

Neo4j에는 RDB의 `ALTER TABLE ... ROLLBACK` 같은 DDL 롤백이 없습니다.
따라서 롤백은 **역방향 마이그레이션 스크립트**로 구현합니다:

```cypher
// ================================================================
// Rollback: v2.1 → v2.0
// ================================================================

// --- 1. 관계 복원 ---
MATCH (a:Concept {id: 'deep_learning'})-[r:DEPENDS_ON]->(b:Concept {id: 'gpu'})
MERGE (a)-[:RELATED_TO]->(b)
DELETE r;

// --- 2. 속성 제거 ---
MATCH (c:Concept)
REMOVE c.category;

// --- 3. 인덱스 제거 ---
DROP INDEX concept_category IF EXISTS;
```

> ⚠️ **프로덕션에서의 롤백 주의**
>
> 노드/관계 **삭제**가 포함된 마이그레이션은 롤백이 불완전할 수 있습니다.
> 삭제된 데이터는 복원할 수 없으므로, 삭제 전 반드시 **neo4j-admin dump**로 백업하세요:
> ```powershell
> docker exec neo4j-ontology neo4j-admin database dump neo4j --to-path=/data/backup/
> ```

## 12.6 Cypher 작성 규칙의 진화

사이클을 반복하면서 **Cypher 작성 규칙**이 점점 강화됩니다.
각 규칙에는 "왜 이 규칙이 생겼는가?"의 실패 경험이 담겨 있습니다.

### v1 규칙 (초기)

```
1. MERGE 사용 (CREATE 금지)
2. id 속성 필수
3. 섹션별 주석 구분
```

### v2 규칙 (Cycle 1 이후 추가)

```
4. Domain.name은 SET하지 않는다
   ← 이슈 #1: 에이전트가 DB→"Database"로 덮어씀

5. 시드 id 참조 필수 (Top 50 목록 확인)
   ← 이슈 #3: agile vs agile_methodology 중복

6. 적재 후 교차 라벨 검증 필수
   ← 이슈 #2: function_point가 Concept + Method 동시 존재

7. 작은따옴표 내 아포스트로피는 큰따옴표로 감싸기
   ← Cypher 구문 오류: "Metcalfe's Law" → 파싱 에러
```

### v3 규칙 (Cycle 2 이후 추가 예정)

```
8. MENTIONED_IN 관계는 문서별 마지막에 일괄 생성
   ← 누락 방지

9. ON CREATE SET / ON MATCH SET 분리
   ← 시드 속성 보호

10. 파일 헤더에 노드/관계 수 기록
    ← 적재 검증 자동화
```

> 💡 **규칙이 늘어나는 것은 좋은 신호**
>
> 규칙이 10개가 넘으면 "너무 복잡하지 않나?"라는 생각이 들 수 있습니다.
> 하지만 각 규칙은 **실패에서 배운 교훈**입니다.
> 규칙이 없으면 같은 실수를 반복하게 됩니다.
>
> 핵심은 규칙을 **문서화하고 프롬프트에 포함**하는 것입니다.
> Claude Code에 규칙 목록을 전달하면, AI가 규칙을 준수한 Cypher를 생성합니다.

## 12.7 [안티패턴] 반복순환에서 흔한 실수

### 한 번에 너무 많이 바꾸기

```
❌ "이슈 5개를 발견했으니 노드 타입 3개 추가 + 관계 5개 변경 + 속성 10개 추가"
   → 어떤 변경이 어떤 효과를 냈는지 추적 불가능
   → 새로운 문제 발생 시 원인 특정 불가

✅ "이번 사이클에서는 이슈 #1, #2만 해결. #3~#5는 다음 사이클"
   → 변경의 영향을 명확히 추적 가능
```

### Before/After 기록 생략

```
❌ "어차피 고쳤으니 됐지"
   → 3개월 후 같은 이슈 재발 시 "이전에 어떻게 해결했더라?" 기억 못함
   → 팀원에게 스키마 변경 이력 전달 불가

✅ 매 사이클마다 정량적 Before/After + 학습 사항 기록
   → 재발 시 즉시 참조, 온보딩 자료로 활용
```

### 검증 없이 다음 사이클 진입

```
❌ "10개 더 적재했으니 바로 50개로 가자"
   → 10개에서 발생한 구조 문제가 50개에서 10배로 증폭
   → 수정 비용이 기하급수적 증가

✅ "10개 적재 → L1~L5 전수 검증 → 이슈 0 확인 → 그 다음 50개"
   → 작은 규모에서 문제를 잡아야 큰 규모에서 편함
```

> 📌 **규칙**: "한 사이클에서 하나의 종류의 변경만. 반드시 검증 후 다음 사이클."

## 12.8 자동화 로드맵

사이클을 반복하면서 수동 작업을 점진적으로 자동화합니다:

### 현재 상태 (Cycle 1 완료)

```
                    자동         반자동        수동
DOC/DOCX → JSON:   ████████████                        (✅ Python 파서)
JSON → Cypher:                   ████████████           (🔄 Claude Code)
Cypher → Neo4j:    ████████████                        (✅ cypher-shell)
검증 쿼리:                                    ████████ (👤 수동 실행)
```

### 목표 상태 (Cycle 4)

```
                    자동         반자동        수동
DOC/DOCX → JSON:   ████████████                        (✅ 유지)
JSON → Cypher:     ██████████   ██                      (🔄 자동 + 수동 검토)
Cypher → Neo4j:    ████████████                        (✅ 유지)
검증 쿼리:          ██████████   ██                      (🔄 자동 + 수동 확인)
```

**자동화 우선순위:**

| 우선순위 | 항목 | 자동화 방법 | 효과 |
|---------|------|-----------|------|
| 1 | 검증 쿼리 | Python 스크립트 (neo4j Driver) | 검증 시간 30분→2분 |
| 2 | MENTIONED_IN 생성 | JSON→Cypher 규칙 기반 | 누락 0건 |
| 3 | BELONGS_TO 일괄 연결 | 도메인 코드 기반 자동 | 누락 0건 |
| 4 | 교차 라벨 검증 | 자동 검출 + 알림 | 조기 발견 |
| 5 | JSON→Cypher 변환 | Claude Code 배치 프롬프트 | 처리 속도 향상 |

## 12.9 [기업 박스] 지속적 스키마 개선 프로세스

기업 환경에서는 반복순환이 **지속적 프로세스**가 됩니다:

```
Sprint 1: 핵심 데이터 500건 → 초기 스키마 → 검증
Sprint 2: 추가 데이터 2,000건 → 스키마 조정 → 검증
Sprint 3: 전체 데이터 10,000건 → 스키마 확정 → 검증
Sprint 4+: 신규 데이터 지속 유입 → 모니터링 → 필요 시 진화
```

**기업 고유 고려사항:**

| 항목 | 개인 프로젝트 | 기업 프로젝트 |
|------|------------|------------|
| **변경 프로세스** | 즉시 변경 | 변경 요청 → 리뷰 → 승인 → 적용 |
| **영향 분석** | 나만 영향 | 다수 팀/서비스 영향 |
| **테스트** | 수동 검증 | CI/CD 파이프라인에 검증 쿼리 통합 |
| **문서화** | CLAUDE.md | 공식 스키마 문서 + 변경 이력 |
| **롤백** | 수동 + 재적재 | 자동 백업 + 롤백 스크립트 |

**CI/CD 파이프라인에 검증 쿼리 통합 예시:**

```yaml
# .github/workflows/neo4j-validate.yml (개념적 예시)

  - name: Run Schema Validation

      cypher-shell "MATCH (n) WHERE n.id IS NULL RETURN count(n)" | assert-zero.py

      # L3: 관계 검증
      cypher-shell "MATCH (n) WHERE NOT (n)--() RETURN count(n)" | assert-zero.py

      # L4: 의미 검증
      cypher-shell "$(cat queries/cross-label-check.cypher)" | assert-zero.py
```

## [실습] 현재 스키마에서 개선점 3개 도출 → 수정 계획 작성

### 목표

현재 적재된 그래프에서 검증을 실행하고,
개선이 필요한 항목 3개를 식별하여 다음 사이클의 수정 계획을 작성합니다.

### 순서

**① 현재 상태 스냅샷 (Neo4j Browser):**

```cypher
// 노드/관계 전체 통계
CALL apoc.meta.stats() YIELD nodeCount, relCount, labels, relTypes
RETURN nodeCount, relCount, labels, relTypes;
```

**② L1~L5 검증 쿼리 실행:**

```cypher
// L2: 필수 속성 NULL 검사
MATCH (c:Concept)
WHERE c.name IS NULL OR c.name_kr IS NULL OR c.domain IS NULL
RETURN c.id, c.name, c.name_kr, c.domain;

// L3: 고립 노드 검색
MATCH (n)
WHERE NOT (n)--()
RETURN labels(n), n.id, n.name
ORDER BY labels(n);

// L4: 교차 라벨 중복 검출
// ※ id()는 Neo4j 5+에서 deprecated → elementId() 사용 권장
MATCH (a), (b)
WHERE a.id = b.id AND elementId(a) < elementId(b)
RETURN labels(a), labels(b), a.id;
```

**③ 이슈 기록:**

발견된 이슈를 다음 형식으로 기록합니다:

```markdown
### 이슈 #N: [제목]
- **계층**: L1/L2/L3/L4/L5
- **발견 쿼리**: (사용한 Cypher)
- **발견 건수**: N건
- **예시**: (구체적 데이터)
- **원인 분석**: (왜 발생했는가)
- **수정 Cypher**: (해결 쿼리)
- **재발 방지**: (규칙 추가)
```

**④ 수정 계획 작성:**

3개 이슈에 대해 수정 순서를 결정합니다:

```
수정 순서: 의존성 기반
1. [가장 근본적 이슈 먼저] — 다른 이슈의 원인일 수 있음
2. [건수가 많은 이슈] — 영향 범위 넓음
3. [재발 위험 높은 이슈] — 규칙 강화 필요
```

### 체크리스트

- [ ] 현재 노드/관계 수 기록 (Before 수치)
- [ ] L1~L5 검증 쿼리 각 1개 이상 실행
- [ ] 3개 이상의 이슈 식별 및 기록
- [ ] 이슈별 수정 Cypher 작성
- [ ] 재발 방지 규칙 도출
- [ ] 수정 순서 결정 (의존성 고려)

---

> 📌 **핵심 정리**
>
> 1. **STEP 3→4→5는 선형이 아니라 순환**: 설계→적재→검증→재설계
> 2. **4단계 사이클**: 표본(10) → 안정화(50) → 확정(100) → 전체(2,392)
> 3. **종료 조건**: "이슈 0건"이 아니라 "**신규 이슈 타입 없음**"
> 4. **Before/After 기록**: 정량적 증거가 학습 자료이자 품질 보증
> 5. **Cypher 작성 규칙의 진화**: 실패에서 배운 교훈을 문서화
> 6. **마이그레이션 스크립트의 멱등성**: 몇 번을 실행해도 같은 결과
> 7. Neo4j의 **스키마리스 특성**이 반복순환을 가능하게 함 — RDB였다면 매번 마이그레이션 지옥

| 항목 | 내용 |

**자동화 파이프라인 설계:**

```
┌─────────────┐    ┌──────────────┐    ┌───────────┐    ┌──────────┐
│ JSON 파일    │───→│ 온톨로지 추출 │───→│ Cypher 생성│───→│ 배치 적재 │
│ (2,381개)   │    │ (Python/LLM) │    │ (자동)     │    │ (자동)    │
└─────────────┘    └──────────────┘    └───────────┘    └──────────┘
                          │                                    │
                   ┌──────┴────────┐              ┌───────────┴──────┐
                   │ 규칙 기반 추출 │              │ 적재 후 자동 검증 │
                   │ + LLM 보정    │              │ L1~L3 기본 진단  │
                   └───────────────┘              └──────────────────┘
```

**핵심 자동화 모듈 (tools/ 하위):**

| 모듈 | 역할 | 입력 → 출력 |
|------|------|-------------|
| `clean` | 텍스트 정제 | JSON → 정제된 JSON |
| `analyze` | 정량 분석 | JSON → 통계 리포트 |
| `extract` | 온톨로지 추출 | JSON → 노드/관계 목록 |
| `generate` | Cypher 생성 | 노드/관계 목록 → .cypher 파일 |
| `load` | 배치 적재 | .cypher 파일 → Neo4j 적재 + 검증 |

**배치 적재 전략:**

```python
# 도메인별 순차 처리, 도메인 내 병렬 가능
domains = ['PM','MG','SW','SV','ST','AI','DS','AL','SE','DB','NW','CA','OS']
for domain in domains:
    files = get_parsed_files(domain)
    for batch in chunk(files, size=10):
        cypher = generate_cypher(batch)       # 10개 단위 Cypher 생성
        load_to_neo4j(cypher)                 # cypher-shell로 적재
        run_basic_validation(domain)          # L1~L3 자동 검증
    run_domain_validation(domain)             # 도메인별 종합 검증
    log_completion(domain)                    # 적재 로그 기록
```

**대규모 적재 시 성능 가이드:**

| 규모 | 적재 방법 | 예상 시간 | 주의사항 |
|------|-----------|-----------|----------|
| ~1,000 노드 | MERGE (Cypher 파일) | ~5분 | 현재 방식 유지 |
| ~10,000 노드 | CALL IN TRANSACTIONS | ~30분 | 500 ROWS 단위 |
| ~50,000 노드 | LOAD CSV + MERGE | ~1시간 | CSV 중간 포맷 필요 |
| 50,000+ | neo4j-admin import | ~10분 | 초기 적재만 가능 (빈 DB) |

### 12.6.3 STEP 8: GraphRAG 확장 — 지식그래프에서 지능으로

| 항목 | 내용 |
|------|------|
| **목표** | 그래프 기반 RAG 구현. "질문→그래프 탐색→LLM 답변" 파이프라인 |
| **전제** | STEP 7 완료 (50,000+ 노드), 벡터 임베딩 준비 |
| **기술 스택** | neo4j-graphrag-python + OpenAI/Claude API + Vector Index |

**GraphRAG 3단계 로드맵:**

```
Phase 1: 전문검색 기반 프로토타입 (난이도: 낮음, 1~2시간)
──────────────────────────────────────────────────
- 현재 그래프의 Fulltext Index만으로 구현 가능
- 키워드 → 관련 노드 → 경로 탐색 → 텍스트 조합
- 추가 비용: 없음

Phase 2: 벡터 임베딩 + 하이브리드 검색 (난이도: 중간, 1~2일)
──────────────────────────────────────────────────
- Concept.definition에 벡터 임베딩 추가
- 의미적 유사도 검색 + 그래프 경로 탐색 결합
- 추가 비용: OpenAI Embedding API (~$5/50K 문서)

Phase 3: neo4j-graphrag-python 통합 (난이도: 높음, 3~5일)
──────────────────────────────────────────────────
- 질문 → 벡터 검색 → 서브그래프 추출 → LLM 답변 생성
- 환각(hallucination) 감소: 그래프가 "근거"를 제공
- 추가 비용: LLM API 사용료 (질문당 ~$0.01~0.03)
```

**Phase 1 → 2 전환을 위한 Vector Index 준비:**

```cypher
// 1. definition에 벡터 임베딩 속성 추가 (Python에서 생성 후 적재)
MATCH (c:Concept)
WHERE c.definition IS NOT NULL
SET c.embedding = $vector;   // 1536차원 벡터 (OpenAI ada-002)

// 2. Vector Index 생성
CREATE VECTOR INDEX concept_embedding IF NOT EXISTS
FOR (n:Concept) ON (n.embedding)
OPTIONS {
  indexConfig: {
    `vector.dimensions`: 1536,
    `vector.similarity_function`: 'cosine'
  }
};

// 3. 시맨틱 검색 쿼리
CALL db.index.vector.queryNodes('concept_embedding', 5, $query_vector)
YIELD node, score
WHERE score > 0.8
MATCH (node)-[r*1..2]-(related)
WHERE NOT related:Domain AND NOT related:Document
RETURN node.name_kr AS 개념, node.definition AS 정의,
       score AS 유사도,
       collect(DISTINCT related.name_kr)[..5] AS 관련개념;
```
