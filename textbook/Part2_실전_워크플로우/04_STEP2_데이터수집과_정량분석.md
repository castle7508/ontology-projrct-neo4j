## 4.2 Python 파싱 파이프라인

### 4.2.1 왜 직접 파서를 만드는가

"문서에서 텍스트를 추출하는 건 기존 도구로 충분하지 않나?"
— 충분하지 않습니다. 이유:

| 기존 도구 | 문제 |
|-----------|------|
| `python-docx` 기본 | `doc.paragraphs`만으로는 테이블 셀 내용 누락 |
| `mammoth` | HTML 변환 목적, 구조화된 JSON 출력 불가 |
| `textract` | 설치 복잡, 한글 인코딩 이슈 |
| `tika` | Java 의존, 무겁고 느림 |

우리에게 필요한 것:
1. **Paragraph + Table 셀** 모두에서 텍스트 추출
2. **섹션 구조**(I, II, III, 가, 나, 다) 보존
3. **메타데이터**(토픽명, 분류, 키워드) 자동 추출
4. **표준화된 JSON** 출력 (Pydantic 모델)
5. **DOC 레거시** 파일도 처리 (win32com)

### 4.2.3 데이터 모델 — Pydantic으로 출력 구조 강제

> 💡 **Pydantic이란?**
>
> Python의 데이터 검증 라이브러리입니다. 클래스로 "데이터의 구조"를 정의하면,
> 그 구조에 맞지 않는 데이터가 들어올 때 즉시 에러를 발생시킵니다.
> 예를 들어 `file_id: str`로 선언하면 숫자가 들어올 때 자동으로 문자열로 변환하거나 에러를 냅니다.
> FastAPI, LangChain 등 최신 Python 프로젝트에서 표준처럼 사용됩니다.

파서의 출력을 Pydantic 모델로 정의하여 **일관된 JSON 구조를 강제**합니다:

```python
class Section(BaseModel):
    """문서 섹션"""
    heading: str = ""     # "I. 인공지능의 개념"
    level: int = 0        # 1 (대제목), 2 (중제목)
    content: str = ""     # 본문 텍스트
    items: list[str] = [] # "가.", "나." 항목 리스트

class TableData(BaseModel):
    """테이블 데이터"""
    headers: list[str] = []       # 첫 행 (헤더)
    rows: list[list[str]] = []    # 나머지 행

class ParsedDocument(BaseModel):
    """파싱된 문서 — 파이프라인의 공식 출력 포맷"""
    file_id: str              # "AI_001"
    filename: str             # "AI_001_인공지능.docx"
    domain: str               # "AI"
    format: str               # "docx" | "doc"
    sections: list[Section]   # 구조화된 섹션
    tables: list[TableData]   # 테이블 데이터
    raw_text: str             # 전체 텍스트 (검색/분석용)
    metadata: dict[str, str]  # 메타데이터 (토픽명, 분류, 키워드)
```

> [설계 원칙] **"파서의 출력 포맷이 이후 모든 단계의 입력 포맷이다."**
>
> Pydantic 모델을 사용하면:
> - JSON 직렬화/역직렬화가 자동 (`model_dump_json()`)
> - 필수 필드 누락 시 즉시 에러
> - 타입 검증 자동 (문자열에 숫자가 들어가면 에러)
> - 후속 단계(온톨로지 추출)에서 구조를 보장

### 4.2.4 DOCX 파서의 핵심 — 테이블 셀 텍스트 추출

> [안티패턴] **Paragraph만 추출하면 데이터의 70%를 놓친다**
>
> 우리 학습 문서는 본문 대부분이 테이블 셀 안에 있습니다.
> `doc.paragraphs`만으로 파싱하면 "빈 문서"처럼 보이는 결과가 나옵니다.
> 실제로 초기 파서 버전에서 이 실수를 했고, "파싱 결과가 왜 이렇게 짧지?"라는
> 문제를 디버깅하면서 테이블 셀 추출의 필요성을 발견했습니다.

일반적인 DOCX 파싱:

```python
# ❌ 이것만으로는 부족
for para in doc.paragraphs:
    print(para.text)
# → 테이블 안의 텍스트가 전부 빠짐!
```

우리 파서의 접근:

```python
# ✅ Paragraph + Table 셀 모두 추출
# 1단계: Paragraph 파싱 (제목, 본문)
for para in doc.paragraphs:
    text = para.text.strip()
    # 스타일(Heading 1, 2, 3)로 섹션 구분
    # 로마숫자(I, II, III)나 한글(가, 나, 다)로 항목 구분

# 2단계: Table 파싱 (본문 핵심!)
for table in doc.tables:
    for row in table.rows:
        for cell in row.cells:
            cell_text = cell.text.strip()
            # 병합 셀 중복 제거 (docx는 병합 셀을 여러 번 반환)
            # 테이블 셀 텍스트도 raw_text에 추가
```

**병합 셀 중복 제거**가 까다로운 포인트입니다:

```python
# 병합된 셀은 python-docx가 같은 텍스트를 여러 번 반환
# 셀의 XML element + 텍스트 조합으로 중복 식별
cell_key = f"{id(table)}_{cell._element.xml[:50]}_{cell_text[:30]}"
if cell_key not in seen_cells:
    seen_cells.add(cell_key)
    cells.append(cell_text)
```

### 4.2.5 DOC 레거시 파일 처리

DOC 파일은 DOCX와 달리 XML 기반이 아닌 바이너리 형식입니다.
`python-docx`로 읽을 수 없으므로 별도 전략이 필요합니다:

| 환경 | 방법 | 라이브러리 |
|------|------|-----------|
| Windows | COM 자동화 | `pywin32` (win32com) |
| Linux/Mac | CLI 도구 | `antiword` |

**Windows COM 자동화 핵심 설계 — 싱글턴 Word 인스턴스:**

```python
# Word COM 인스턴스를 모듈 레벨 싱글턴으로 유지
_word_app = None

def _get_word_app():
    """Word 인스턴스 재사용 (성능 최적화)"""
    global _word_app
    if _word_app is not None:
        return _word_app
    _word_app = win32com.client.Dispatch("Word.Application")
    _word_app.Visible = False
    return _word_app
```

> 💡 **왜 싱글턴인가?**
>
> DOC 파일 1개당 Word 인스턴스를 생성하면:
> - 2,392개 파일 × Word 시작 시간(~2초) = **약 80분** 낭비
> - 메모리 폭발 위험
>
> 싱글턴으로 하나의 Word 인스턴스를 재사용하면:
> - Word 시작 1회 + 파일 열기/닫기만 반복
> - 전체 처리 시간 **~15분** (5배 이상 단축)

### 4.2.6 파일 ID 추출 규칙

파일명에서 도메인 코드와 문서 ID를 추출합니다:

```python
def extract_file_id(filename: str) -> tuple[str, str]:
    """
    CA_001_멧칼프 법칙.docx → ("CA", "CA_001")
    AI_047.5_강화학습.docx  → ("AI", "AI_047.5")
    """
    match = re.match(r"^([A-Z]{2})_(\d+(?:\.\d+)?)", stem)
```

**특수 케이스**: 일부 문서는 소수점 번호를 가집니다 (`AI_047.5`).
이는 기존 문서 사이에 추가 문서를 삽입한 것으로,
정규표현식에서 `\d+(?:\.\d+)?`로 처리합니다.

## 4.4 정량 분석 — 스키마 설계의 과학적 근거

### 4.4.1 왜 정량 분석을 먼저 하는가

스키마 설계의 가장 흔한 실수:

```
❌ "나는 IT를 잘 아니까, 대충 이런 구조면 되겠지"
   → 주관적 설계 → 데이터와 불일치 → 대규모 수정 필요

✅ "데이터가 이렇게 말하고 있으니, 이런 구조가 필요하다"
   → 데이터 기반 설계 → 실제와 일치 → 안정적 확장
```

> [설계 원칙] **"데이터가 스키마를 결정한다"**
>
> 스키마는 설계자의 머릿속이 아니라, **데이터의 특성**에서 나와야 합니다.
> - 어떤 **키워드**가 가장 많이 등장하는가? → 시드 노드 후보
> - 어떤 **도메인**이 자주 교차하는가? → 공유 노드 필요성
> - 어떤 **문제 유형**이 출제되는가? → 관계 타입 결정

### 4.4.2 분석 대상과 방법

기출문제 데이터(`documents/문제.txt`)를 전수 분석합니다:

```
분석 원본:
- 22,518줄
- 추정 문제 수: 9,291개
- 기술사 기출 + 모의 + Final Review 포함
```

분석 항목:

| 분석 | 방법 | 산출물 |
|------|------|--------|

### 4.4.5 도메인 교차 분석 → 공유 노드 필요성

같은 키워드가 **여러 도메인에서 동시에 출제**되는 패턴을 분석합니다:

| 교차 도메인 | 교차 문제 수 | 대표 키워드 |
|------------|-------------|------------|
| OS_CA + SW | 62 | 프로세스, 메모리, 스케줄링 |
| DB + SW | 42 | 트랜잭션, 정규화, 인덱스 |
| AI + SE | 34 | 취약점, 개인정보보호 |
| SE + SV | 34 | 인증, 클라우드 보안 |
| NW + SE | 33 | 방화벽, VPN, 침입탐지 |
| AI + OS_CA | 31 | GPU, 병렬처리 |

**스키마 설계 시사점:**

> [설계 원칙] **"도메인 교차 = 공유 노드"**
>
> "프로세스"가 OS에서도, SW에서도 출제된다면,
> 두 도메인에 각각 별도 노드를 만들 것인가? 아니면 **하나의 노드를 공유**할 것인가?
>
> 답: **공유 노드 + BELONGS_TO 다중 연결**.
> ```cypher
> // 프로세스는 OS와 SW 두 도메인에 모두 소속
> MERGE (p:Concept {id: 'process'})
> SET p.domain = 'OS',                      // 주 도메인
>     p.cross_domains = ['OS', 'SW', 'PM']  // 교차 도메인
> MERGE (p)-[:BELONGS_TO]->(:Domain {id: 'OS'})
> ```
>
> 이것이 **MERGE 기반 성장**의 핵심입니다:
> OS 도메인 문서에서 `process`를 MERGE하면 노드가 생성되고,
> SW 도메인 문서에서도 같은 `process`를 MERGE하면 **기존 노드에 연결**됩니다.
> 도메인 간 연결이 자동으로 형성됩니다.

교차 빈도 상위 분석에서 또 하나의 패턴이 보입니다:

```
8개 도메인 동시 출제 키워드: 11건
→ "보안"과 "품질" 관련 키워드는 거의 모든 도메인에서 등장
→ 이러한 범용 Concept은 시드 노드로 반드시 포함
```

### 4.4.6 문제 유형 분포 → 관계 패턴 결정

기출문제의 출제 형식을 분류합니다:

| 문제 유형 | 문제 수 | 비율 | 예시 |
|----------|---------|------|------|
| **단답설명형** | 3,906 | 42.0% | "~를 설명하시오" |
| **서술형(3항목)** | 1,377 | 14.8% | "~의 개념, 유형, 적용 방안" |
| **비교형** | 771 | 8.3% | "A와 B를 비교 설명하시오" |
| **코드형** | 301 | 3.2% | "~를 구현하시오" |
| **계산형** | 205 | 2.2% | "~를 계산하시오" |
| **서술형(2항목)** | 184 | 2.0% | "~의 개념과 특징" |
| **다이어그램형** | 152 | 1.6% | "~의 구조를 그리시오" |
| **서술형(4항목)** | 125 | 1.3% | "~의 개념, 구성, 절차, 사례" |
| 기타 | 2,270 | 24.4% | 복합형, 사례형 등 |

**스키마 설계 시사점 — 문제 유형이 관계 타입을 결정합니다:**

| 문제 유형 | 필요한 관계 | 근거 |
|----------|-----------|------|
| "~를 설명하시오" (42%) | `definition` 속성 | 정의 저장 필요 |
| "~의 유형을 설명하시오" | `HAS_TYPE`, `SUBCLASS_OF` | 계층/분류 관계 |
| "A와 B를 비교하시오" (8.3%) | `COMPARED_WITH` | 비교 관계 + perspectives 속성 |
| "~의 구성요소" | `HAS_COMPONENT` | 부분-전체 관계 |
| "~의 절차/단계" | `HAS_PHASE` | 순서 관계 |
| "~의 대응방안" | `THREATENS`, `COUNTERED_BY` | 위협-대응 쌍 |
| "~를 구현한 기술" | `IMPLEMENTS`, `USES` | 구현/활용 관계 |

> 💡 **비교형이 8.3%나 된다는 것의 의미**
>
> 771개 문제가 "A와 B를 비교"하는 형식입니다.
> 이는 `COMPARED_WITH` 관계가 반드시 필요하고,
> 비교 관점(perspectives)을 관계 속성으로 저장해야 한다는 근거입니다.
> ```cypher
> MERGE (a:Technology {id: 'aes'})
> MERGE (b:Technology {id: 'des'})
> MERGE (a)-[:COMPARED_WITH {
>   perspectives: ['키 길이', '블록 크기', '라운드 수', '보안 강도']
> }]->(b);
> ```

## 4.5 [기업 박스] 기업 데이터 수집 전략

기업 환경에서는 데이터 소스가 더 다양합니다:

| 데이터 소스 | 수집 방법 | 난이도 | 예시 |
|------------|----------|--------|------|
| **RDB** | SQL 쿼리 → CSV → LOAD CSV | 낮음 | 제품 카탈로그, 고객 DB |
| **API 로그** | JSON/NDJSON → apoc.load.json | 중간 | 서비스 호출 패턴 |
| **위키/문서** | 크롤링 → 텍스트 추출 → JSON | 중간 | Confluence, Notion |
| **도메인 전문가** | 인터뷰 → 구조화 | 높음 | 암묵지 → 형식지 전환 |
| **기존 온톨로지** | RDF/OWL → n10s 임포트 | 중간 | 산업 표준 온톨로지 |

**기업 데이터 수집의 3대 원칙:**

1. **점진적 수집**: 전체 데이터를 한 번에 수집하지 말 것. 핵심 10%부터 시작
2. **표준화된 중간 포맷**: 어떤 소스든 JSON으로 표준화한 후 파이프라인에 투입
3. **메타데이터 보존**: 원본 출처, 수집 일시, 담당자를 반드시 기록 (데이터 리니지)

**정량 분석의 기업 적용:**

| 우리 프로젝트 | 기업 적용 |
|-------------|----------|
| 기출문제 키워드 빈도 | 검색 로그 키워드 빈도 |
| 도메인 교차 출제 | 부서 간 공유 데이터 |
| 문제 유형 분포 | 비즈니스 질문 유형 분포 |
| Top 50 시드 | 핵심 비즈니스 엔티티 Top N |

> 💡 **기업에서 정량 분석의 ROI**
>
> "데이터가 1만 건인데 정량 분석까지 해야 하나?"
> → 해야 합니다. 1만 건을 감으로 설계하면 STEP 5(검증)에서 30% 이상 수정이 발생합니다.
> 정량 분석에 하루를 투자하면, 검증/수정에 일주일을 절약합니다.
