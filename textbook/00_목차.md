### Part 1: 기초

#### Ch.01 — 그래프DB와 Neo4j 입문 (~30p)

```
1.1 데이터의 본질은 "관계"다
    - 현실 세계의 데이터는 테이블이 아니라 네트워크
    - 예시: "딥러닝은 머신러닝의 하위 분야이고, CNN은 딥러닝을 구현하며,
            이미지 분류에 사용되고, GPU에 의존한다"
    - 이 한 문장을 RDB로 표현하면? → 4개 테이블 + 3개 JOIN
    - 같은 문장을 그래프로 표현하면? → 4개 노드 + 4개 관계 (직관적)

1.2 그래프 데이터베이스 핵심 개념
    1.2.1 노드 (Node)
        - 엔티티를 표현하는 기본 단위
        - 라벨(Label)로 유형 분류: (:Concept), (:Technology)
        - 하나의 노드에 여러 라벨 가능
    1.2.2 관계 (Relationship)
        - 노드 간 연결, 반드시 방향이 있음
        - 타입(Type)으로 의미 부여: -[:SUBCLASS_OF]->
        - 관계에도 속성 부여 가능
    1.2.3 속성 (Property)
        - 키-값 쌍으로 노드/관계에 메타데이터 저장
        - 지원 타입: 문자열, 숫자, 불리언, 배열, 시간, 공간
    1.2.4 경로 (Path)
        - 노드와 관계의 연속된 시퀀스
        - 그래프의 진정한 힘 = 경로 탐색

1.3 Property Graph vs RDF — 왜 Neo4j를 선택했나
    - Property Graph Model: 직관적, 속성 풍부, 쿼리 유연
    - RDF/OWL: 표준화, 추론 엔진, 시맨틱웹
    - 선택 기준: "구축 속도 + 쿼리 유연성" → Property Graph
    - [기업 박스] RDF가 더 적합한 경우 (규제, 표준 교환)

1.4 Neo4j 에코시스템 한눈에
    1.4.1 Neo4j 에디션 (Community vs Enterprise)
    1.4.2 핵심 플러그인 (APOC, GDS, n10s)
    1.4.3 클라이언트 도구 (Browser, Bloom, Aura)

1.5 RDB vs 그래프DB — 실전 비교
    - 같은 질문 5개를 SQL과 Cypher로 풀어보기
    - Q1: "암호화의 하위 기술을 모두 찾아라" (계층 탐색)
    - Q2: "AES와 DES를 비교하시오" (관계 속성 조회)
    - Q3: "보안 도메인 출제 Top 10" (집계)
    - Q4: "딥러닝과 3-hop 이내 모든 개념" (경로 탐색)
    - Q5: "두 개념의 공통 상위 개념" (최단경로)

    [실습] Neo4j Browser에서 Q1~Q5 직접 실행해보기
```

#### Ch.02 — Cypher 핵심문법 30분 완성 (~25p)

```
2.1 이 챕터의 목적
    - Part2 워크플로우를 따라가기 위한 "최소 Cypher"
    - 깊이 있는 문법은 Part3(Ch.13~15)에서 다룸
    - "읽을 수 있고, 기본 쿼리를 쓸 수 있는" 수준 목표

2.2 읽기 — MATCH + RETURN
    - 기본 패턴, 관계 패턴, 필터링, 별칭과 정렬

2.3 쓰기 — CREATE vs MERGE
    - CREATE: 무조건 생성 (중복 위험!)
    - MERGE: 있으면 재사용, 없으면 생성 (온톨로지의 핵심)
    - ON CREATE SET / ON MATCH SET
    - [설계 원칙] "온톨로지에서는 항상 MERGE"

2.4 수정과 삭제
    - SET, REMOVE, DELETE, DETACH DELETE

2.5 제약조건과 인덱스 (맛보기)
    - UNIQUE 제약조건, 기본 인덱스

2.6 [실습] 미니 온톨로지 만들기
    - 3개 Concept + 2개 관계 + 1개 Domain
    - MERGE로 생성 → MATCH로 조회 → SET으로 수정 → 검증
```

#### Ch.05 — STEP 3: 스키마 설계 - 온톨로지 모델링 ★★★ (~55p)

```
5.1 온톨로지란 무엇인가
    5.1.1 철학에서 정보과학으로
        - 아리스토텔레스의 존재론 → 인공지능의 지식 표현
        - Tom Gruber의 정의: "공유된 개념화의 명시적 명세"
    5.1.2 온톨로지의 4계층
        ┌──────────────────────────────────────┐
        │ Level 4: 형식 온톨로지 (OWL/RDF)     │
        │ Level 3: 도메인 온톨로지 ← 우리 수준  │
        │ Level 2: 택소노미 (계층 분류)         │
        │ Level 1: 용어 사전 (시소러스)         │
        └──────────────────────────────────────┘
    5.1.3 온톨로지 vs 택소노미 vs 시소러스 vs 스키마
    5.1.4 Property Graph에서의 온톨로지
        - OWL/RDF vs Property Graph 세계관
        - 트레이드오프: 추론 엔진 vs 쿼리 유연성

5.2 Question-Driven Modeling (QDM) 방법론
    5.2.1 핵심 철학
        - "좋은 스키마 = 좋은 질문에 답할 수 있는 구조"
        - [설계 원칙] "답변할 수 없는 질문이 있다면 스키마가 부족하다"
    5.2.2 QDM 5단계 프로세스
        ① 핵심 질문 도출
        ② 답변 패턴 분석
        ③ 노드 후보 식별
        ④ 관계 패턴 도출
        ⑤ 속성 확정
    5.2.3 QDM 실전 적용 — 우리 프로젝트
        - 질문 예시 10개 → 패턴 분석 → 스키마 도출 과정 전체 재현
        - Q: "딥러닝의 하위 기술은?" → SUBCLASS_OF
        - Q: "AES와 DES를 비교하시오" → COMPARED_WITH + perspectives
        - Q: "SDLC의 단계를 설명하시오" → HAS_PHASE
        - Q: "DDoS 공격의 대응방안은?" → THREATENS + COUNTERED_BY
        - Q: "ISO 27001이 정의하는 보안 요소는?" → DEFINED_BY
        - ... (10개 전체)
    5.2.4 [기업 박스] 비즈니스 질문에서 QDM 적용

5.3 모델링 접근법 비교
    5.3.1 Top-Down: 도메인 지식에서 출발
    5.3.2 Bottom-Up: 데이터에서 출발
    5.3.3 Middle-Out: QDM의 접근
    5.3.4 우리 프로젝트: Bottom-Up + Middle-Out 하이브리드

5.4 스키마 설계의 보편 원칙
```
